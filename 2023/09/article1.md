# 靜態 vs 動態型別

## 型別檢查的時機

### 核心差異

靜態型別（static typing）和動態型別（dynamic typing）的根本差異在於**型別檢查的時機**。

**靜態型別**：型別在編譯期檢查。變數的型別必須在編譯時確定，型別錯誤在執行前就被捕獲。

```python
# TypeScript（靜態）
function greet(name: string): string {
    return "Hello, " + name;
}
// 編譯器會發現：greet(42) // 錯誤！
```

**動態型別**：型別在執行期檢查。變數可以指向任何型別的資料，型別錯誤只在執行時才會出現。

```python
# Python（動態）
def greet(name):
    return "Hello, " + name
# 執行時才會錯誤
# greet(42)  # TypeError: can only concatenate str (not "int") to str
```

### 各語言的型別定位

| 語言 | 型別系統 | 特點 |
|------|---------|------|
| Python | 動態 | 3.5+ 支援型別提示（僅提示，非強制） |
| TypeScript | 靜態 | 完全相容 JavaScript，漸進式採用 |
| Java | 靜態 | 編譯期檢查，執行時期也有型別資訊 |
| Ruby | 動態 | 純物件導向，一切皆物件 |
| Rust | 靜態 | 所有權系統 + 強型別 |
| JavaScript | 動態 | TypeScript 是靜態超集 |

### 靜態型別的優點

1. **早期錯誤發現**：型別錯誤在編譯時被捕捉
2. **編譯器最佳化**：型別資訊允許編譯器生成更高效的機器碼
3. **更好的工具支援**：IDE 可以精確地進行自動補全和重構
4. **自我文件化**：型別宣告本身就是文件

### 動態型別的優點

1. **快速原型開發**：不需要宣告型別，寫作速度更快
2. **鴨子型別（duck typing）**：只要物件有特定方法就能使用，無需繼承
3. **元程式設計更容易**：動態建立和修改程式結構更方便
4. **學習曲線更平緩**：不需要理解型別系統的概念

### 漸進型別（Gradual Typing）

現代語言的趨勢是**漸進型別**——在同一個語言中混合靜態和動態檢查：

- **TypeScript**：JavaScript + 型別註記，可逐步採用
- **Python**：型別提示僅供檢查工具使用，直譯器不強制
- **Hack**（Facebook）：PHP 的靜態型別版本

### 選擇策略

專案類型與型別系統的選擇：

- **大型專案**：靜態型別提供更好的可維護性
- **小型腳本**：動態型別的開發速度更快
- **核心函式庫**：靜態型別確保介面的正確性
- **資料科學**：動態型別更適合探索性編程

### 實務建議

對於新專案，建議從靜態型別語言開始（TypeScript 或 Rust），讓編譯器幫你確保型別正確性。對於原型開發或資料科學探索，Python 的動態型別能提供更高的迭代速度。許多團隊採用「核心用靜態型別、腳本用動態型別」的策略。
