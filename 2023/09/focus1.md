# 程式語言設計原則

## 語法、語意與抽象化

### 前言

程式語言設計是一門平衡的藝術。設計者必須在表達力、簡單性、效能和安全性之間取得取捨。本期將探討程式語言設計的核心原則。

### 語法與語意

**語法（Syntax）** 定義程式的表面結構——關鍵字、運算子、括號的排列方式。**語意（Semantics）** 定義程式的行為——每個語法結構的執行效果。

好的語法應該讓常見模式簡潔表達（如 Python 的 `for x in list`），而語意則需要精確無歧義（如 Rust 的所有權規則）。

### 抽象化機制

程式語言提供多層次的抽象化：

- **函數抽象**：將運算序列封裝為可重複呼叫的單元
- **資料抽象**：將資料結構的內部表示隱藏起來（封裝）
- **控制抽象**：將控制流程封裝為高階結構（如 `with`、`using`）

### 設計取捨

**靜態 vs 動態檢查**：早期綁定（編譯期檢查）提供安全性但降低靈活性；晚期綁定（執行時期檢查）增加靈活性但可能引入錯誤。

**表達力 vs 簡單性**：Perl 的「TMTOWTDI」（多種方式做同一件事）增加表達力但也增加學習曲線。Python 的「只有一種明顯的方式來做」降低認知負擔。

### 正規語法方法

現代語言設計常用 **BNF（Backus-Naur Form）** 描述語法：

```python
expr ::= term ('+' term)*
term ::= factor ('*' factor)*
factor ::= '(' expr ')' | NUMBER
```

### 經典原則

- **正交性**：語言特性應能自由組合，無意外互動。例如 C++ 中 const 與 reference 的組合行爲不一致就是違反正交性的例子。
- **最小驚訝原則**：語言行爲應符合程式設計師的直覺。Python 的 `for x in list` 就比 C 的 `for(i=0;i<n;i++)` 更符合自然語言習慣。
- **安全優先**：能讓編譯器捕獲的錯誤就不該留到執行時期。Rust 的所有權系統在編譯期就杜絕了記憶體錯誤，這是安全優先原則的極致體現。
- **零成本抽象**：不使用的功能不應產生開銷，且使用抽象不應比手寫程式碼產生更多開銷。這是 C++ 和 Rust 的核心哲學——你只為你使用的功能付費。

### 語法糖與語言演進

好的語言設計會逐步吸收社群中常見的模式作為語法糖。例如 Python 的裝飾器（`@decorator`）最初只是一個 PEP 提案，後來成為語言的核心特性。Rust 的 `?` 運算符將錯誤傳播模式從樣板程式碼變為語言內建操作。

### 知名語言的設計哲學

- **Python**：「可讀性至上」，強制縮排、簡潔語法
- **Rust**：「安全與效能並重」，所有權系統、零成本抽象
- **Go**：「簡單為王」，只有一種風格、快速編譯
- **Haskell**：「純粹的數學美」，純函數式、惰性求值
