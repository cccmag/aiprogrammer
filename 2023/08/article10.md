# 編譯器與直譯器的差異

## 前言

編譯器（Compiler）和直譯器（Interpreter）是執行高階語言的兩種主要方式。雖然兩者的目標都是執行開發者撰寫的程式碼，但它們在架構、效能和開發體驗上有顯著差異。本文深入分析兩者的設計取捨。

## 定義

### 編譯器

編譯器將原始碼一次性翻譯為目標語言（通常是機器碼），然後獨立執行產生的目標碼。

```
原始碼 → 編譯器 → 目標碼 → 載入器 → 執行
```

### 直譯器

直譯器逐條讀取、分析並執行原始碼，不產生獨立的目標檔案。

```
原始碼 → 直譯器（分析+執行）
       ↓
     執行結果
```

## 純編譯 vs 純直譯

### 純編譯的範例：C

```c
// source.c
#include <stdio.h>
int main() {
    printf("Hello\n");
    return 0;
}
```

```bash
$ gcc source.c -o program  # 編譯（產生可執行檔）
$ ./program                 # 執行
Hello
```

### 純直譯的範例：Shell

```bash
#!/bin/bash
echo "Hello"
```

```bash
$ ./script.sh   # 直接執行，無需編譯步驟
Hello
```

## 混合模型

### 位元組碼直譯：Java, Python

許多現代語言採用折衷方案：

**Java（JVM）：**
```
原始碼 (.java) → javac 編譯 → 位元組碼 (.class) → JVM 直譯/JIT → 執行
```

**Python：**
```
原始碼 (.py) → 編譯（透明）→ 位元組碼 (.pyc) → PVVM 直譯 → 執行
```

### Just-In-Time 編譯：Java JIT, JavaScript V8

JIT（即時編譯）是編譯和直譯的混合體：

```
1. 直譯器先執行（快速啟動）
2. 監控熱點代碼（經常執行的部分）
3. JIT 編譯器將熱點代碼編譯為機器碼
4. 後續執行直接使用機器碼（高效）
```

## 架構差異

### 兩階段的處理

編譯器將程式處理分為分析階段和綜合階段：

1. **分析階段（前端）**：詞法分析、語法分析、語意分析
2. **綜合階段（後端）**：最佳化、程式碼生成

直譯器則將分析與執行交織在一起：

```
while (有程式碼) {
    讀取一行
    分析這一行
    執行這一行
}
```

## 優缺點比較

| 比較維度 | 編譯器 | 直譯器 |
|---|---|---|
| 啟動速度 | 慢（需先編譯） | 快 |
| 執行速度 | 快（機器碼執行） | 慢（逐行分析） |
| 記憶體使用 | 編譯後較少 | 需要攜帶直譯器 |
| 錯誤回報 | 一次性全部 | 執行到才發現 |
| 除錯體驗 | 需要符號表（DWARF） | 天然互動式 |
| 可攜性 | 需針對平台編譯 | 只需直譯器 |
| 安全性 | 機器碼直接執行 | 可加沙箱 |

## 語言設計的影響

語言設計通常會影響執行模型的選擇：

**偏向編譯的語言：**
- C, C++, Rust：追求執行效能
- Go：平衡編譯速度和執行速度
- Swift：重視機器碼品質

**偏向直譯的語言：**
- Python, Ruby：重視開發者體驗
- JavaScript：瀏覽器環境需求
- Bash：互動式指令環境

**混合的語言：**
- Java：位元組碼 + JIT
- C#：位元組碼 + JIT（也支援 AOT）
- Julia：LLVM JIT 編譯

## 歷史的演變

有趣的是，語言的執行模型在歷史中不斷演變：

- **1990s**：Java 剛推出時只是位元組碼直譯器（慢）
- **2000s**：Java HotSpot JIT 徹底改變了效能口碑
- **2010s**：JavaScript V8 的 JIT 讓 Web App 成為可能
- **2020s**：WebAssembly 讓 C/Rust 程式在瀏覽器中接近原生速度

## 結語

編譯器和直譯器不是二元對立的選擇，而是一個連續頻譜。現代語言往往採用混合策略——用編譯器產生可移植的中間表示，用直譯器或 JIT 編譯器來執行。理解兩者的優缺點，有助於開發者選擇適合的工具和設計新的程式語言。
