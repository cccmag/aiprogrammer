# 行程 vs 執行緒

## 基本定義

行程（process）和執行緒（thread）是作業系統中兩個核心的抽象概念。

**行程**是一個正在執行程式的實例。它擁有獨立的位址空間、程式計數器、系統資源（如檔案描述符）和執行狀態。

**執行緒**是行程內部的執行單元。一個行程可以包含多個執行緒，它們共享行程的位址空間和資源，但擁有各自的堆疊和暫存器。

## 核心差異

### 位址空間

行程擁有獨立的虛擬位址空間。核心通過 MMU 和頁表隔離不同行程的記憶體，確保一個行程無法存取另一個行程的資料。

同一行程的多個執行緒共享位址空間——一個執行緒分配的記憶體可以被同一行程的其他執行緒訪問。

### 資源開銷

建立行程需要分配 PCB（Process Control Block）、頁表、檔案描述符表等，開銷較大。

建立執行緒只需要分配 TCB（Thread Control Block）和執行緒堆疊，開銷小得多。

| 項目 | 行程 | 執行緒 |
|------|------|--------|
| PCB/TCB 大小 | ~1-2 KB | ~幾百位元組 |
| 頁表分配 | 需要（多層頁結構） | 不需要（共用） |
| 建立時間 | 數十微秒 | 數微秒 |
| 上下文切換 | 1-100 微秒 | < 1 微秒 |

### 通訊方式

行程間通訊（IPC）需要通過核心提供的機制：

```
管線（Pipe）
訊息佇列（Message Queue）
共享記憶體（Shared Memory）
信號（Signal）
Socket
```

執行緒間通訊直接讀寫共享變數即可——不需要核心介入，但需要使用同步機制避免競爭條件。

### 隔離性

一個行程的崩潰不會影響其他行程——核心會清理終止行程的資源，其他行程繼續正常執行。

一個執行緒的崩潰（如段錯誤）可能終止整個行程，導致所有執行緒都無法繼續執行。

## 適用場景

### 何時使用多行程？

- 需要強隔離性的應用（如瀏覽器不同分頁）
- 需要不同使用者權限的任務
- 需要跨機器擴展的應用

### 何時使用多執行緒？

- 需要共享大量資料的任務
- 需要低延遲通訊的並行元件
- I/O 密集的應用（如 Web 伺服器）
- 需要利用多核心的計算密集任務

## 協程與纖程

除了核心執行緒，現代語言還提供了更輕量的並行抽象：

**協程（Coroutine）**：使用者層級的輕量執行緒，由語言運行時管理（而非作業系統）。切換只需要保存少量暫存器，不需系統呼叫。

- Python：async/await（asyncio）
- Go：goroutine（~4KB 堆疊初始大小）
- Rust：async/await
- C++20：coroutines

**纖程（Fiber）**：更輕量的協作式執行緒，完全由應用程式控制切換時機。Windows 和 Java（Loom）有原生支援。

## 選擇指南

```
需要隔離性高？   → 使用多行程
需要大量共享資料？ → 使用多執行緒
需要大量並行任務？ → 使用協程/goroutine
需要低延遲？    → 使用執行緒或協程
```
