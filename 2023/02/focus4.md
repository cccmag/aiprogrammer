# 執行緒與並行（1980s-2023）

## 執行緒的引入

傳統的 Unix 行程有兩個角色：資源擁有者和執行單元。執行緒的引入將這兩個角色分離：

- **行程（Process）**：資源擁有者——擁有獨立的位址空間、檔案描述符、信號處理器
- **執行緒（Thread）**：執行單元——擁有獨立的堆疊、暫存器、程式計數器，但與同行程的其他執行緒共享位址空間

一個行程至少包含一個執行緒（主執行緒），可以包含多個執行緒。

## 執行緒 vs 行程

| 特性 | 行程 | 執行緒 |
|------|------|--------|
| 位址空間 | 獨立 | 共享 |
| 資源開銷 | 大（PCB、頁表） | 小（僅堆疊和暫存器） |
| 上下文切換 | 慢（需切換頁表） | 快（不切換位址空間） |
| 通訊方式 | IPC（管線、共享記憶體） | 直接讀寫共享變數 |
| 隔離性 | 高（一個行程崩潰不影響其他） | 低（一個執行緒崩潰整個行程終止） |
| 建立時間 | 慢（需要複製位址空間） | 快（僅建立堆疊和 TCB） |

## 執行緒的類型

### 使用者層級執行緒（User-Level Threads, ULT）

完全在使用者空間管理，核心不知道執行緒的存在。

```
優點：不需核心介入，切換極快（函式呼叫），可以在任何 OS 上實作
缺點：一個執行緒阻塞會阻塞整個行程；無法利用多核心
```
代表：GNU Portable Threads、早期 Java 執行緒

### 核心層級執行緒（Kernel-Level Threads, KLT）

由作業系統核心直接管理。

```
優點：可以充分利用多核心；一個執行緒阻塞不影響其他執行緒
缺點：切換需要系統呼叫，開銷較大
```
代表：Windows 執行緒、Linux NPTL

### 混合模型（Hybrid Threads）

結合兩者優點——多個使用者執行緒映射到較少的核心執行緒。

- **多對一**：多個使用者執行緒對應一個核心執行緒
- **一對一**：每個使用者執行緒對應一個核心執行緒（現代主流）
- **多對多**：多個使用者執行緒對應多個核心執行緒（Solaris）

## 執行緒池

執行緒池（Thread Pool）是一種常見的設計模式：

```
執行緒池 = 預先建立的執行緒集合
工作佇列 = 待執行任務的 FIFO 佇列
運作方式 = 執行緒從佇列取出任務執行，執行完取下一任務
```

優勢：
- 避免頻繁建立/銷毀執行緒的開銷
- 限制並行執行緒數量，防止系統過載
- 任務提交與執行解耦

## 工作竊取（Work Stealing）

每個處理器核心維護自己的工作佇列。當某個核心的佇列為空時，它從其他核心的佇列「偷取」任務。

這是 Fork-Join Pool 和 Go 排程器的核心機制。

## 現代語言中的執行緒

| 語言 | 執行緒模型 | 特點 |
|------|-----------|------|
| C (pthreads) | 1:1 核心執行緒 | POSIX 標準，跨平台 |
| C++11+ | std::thread | 跨平台抽象，通常 1:1 |
| Java | java.lang.Thread | JVM 管理，作業系統映射 |
| Python | threading.Thread | GIL 限制，不適合 CPU 密集 |
| Go | goroutine | M:N 排程，輕量（~4KB 堆疊） |
| Rust | std::thread | 1:1，所有權安全 |
| C# | Task/TPL | 工作竊取，非同步支援 |

## Go 的 M:N 排程模型

Go 的 goroutine 是使用者層級輕量執行緒：

- **M**：機器執行緒（核心執行緒）
- **P**：處理器（排程上下文）
- **G**：goroutine

Go 運行時將 G 分配到 P 上執行，P 再綁定到 M。當 G 阻塞時，P 會切換到另一個 M。Go 的排程器是協作式的——goroutine 在系統呼叫或通道操作時自動讓出。
