# CPU 排程演算法（1960s-2023）

## 排程的基本問題

CPU 排程器決定哪個就緒行程獲得 CPU 使用權。排程發生在以下時機：

- **非搶占式（Non-preemptive）**：行程主動釋放 CPU（等待 I/O 或執行結束）
- **搶占式（Preemptive）**：作業系統強制收回 CPU（時間片到期、高優先權行程就緒）

## 排程演算法的評估指標

- **CPU 使用率**：CPU 忙碌的時間比例（理想 40%-90%）
- **輸送量（Throughput）**：單位時間內完成的行程數量
- **周轉時間（Turnaround Time）**：從行程提交到完成的總時間
- **等待時間（Waiting Time）**：行程在就緒佇列中等待的總時間
- **回應時間（Response Time）**：從提交到首次回應的時間（互動式系統重要）

## 經典排程演算法

### 先到先服務（FCFS）

最簡單的演算法：依抵達順序執行，非搶占式。

```
優點：簡單、公平
缺點：短行程被長行程阻塞（護理效應），平均等待時間長
```
**範例**：行程 P1(5)、P2(3)、P3(2)，平均等待時間 = (0 + 5 + 8) / 3 = 4.3

### 最短工作優先（SJF）

選擇服務時間最短的行程執行。

```
優點：平均等待時間最小（理論最優）
缺點：難以預知服務時間；可能導致長行程飢餓
```
**範例**：P1(5)、P2(3)、P3(2)，順序 P3→P2→P1，平均等待時間 = (0 + 2 + 5) / 3 = 2.3

### 優先權排程

每個行程被賦予優先權，優先權最高的行程先執行。

- **靜態優先權**：在行程創建時決定（如系統行程優先權高）
- **動態優先權**：隨時間調整（如老化機制防止飢餓）

### 時間片輪轉（RR）

每個行程獲得一個固定的時間片（time quantum）。就緒行程以循環方式排隊。

```
優點：回應時間短，互動性好
缺點：上下文切換開銷大（時間片太小）；太大則退化成 FCFS

關鍵選擇：時間片大小
- 太小：過多上下文切換，CPU 有效利用率低
- 太大：互動回應變慢
- 典型值：10-100ms
```

### 多層佇列排程

將行程分為不同類別（如前景互動行程、背景批次行程），每個類別使用不同的排程演算法。

```
前景（RR，時間片小）→ 高優先權
背景（FCFS）         → 低優先權
```

### 多層回饋佇列（MLFQ）

改進版的多層佇列——行程可以在佇列之間移動：

- 新行程進入最高優先權佇列
- 用完時間片則降級到下一級
- 長時間未執行則升級（老化）
- 每個佇列可能有不同的時間片大小

MLFQ 是現代通用作業系統（如 Linux、Windows）的主流排程演算法。

## 現代排程器

| 排程器 | 作業系統 | 特點 |
|--------|---------|------|
| O(1) | Linux 2.6 | 固定時間排程，擴展性好 |
| CFS | Linux 2.6.23+ | 完全公平排程，紅黑樹結構 |
| ULE | FreeBSD | 排程器，支援 SMP |
| MQS | Windows NT+ | 多層回饋佇列，32 層優先權 |

## Linux CFS 排程器

CFS（Completely Fair Scheduler）採用「虛擬運行時間」（vruntime）的概念：

- 每個行程記錄已消耗的 CPU 時間
- CFS 選擇 vruntime 最小的行程執行
- 使用紅黑樹（O(log n)）管理就緒行程
- 當行程睡眠時，vruntime 不變，醒來後獲得「補償」
