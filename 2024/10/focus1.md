# 專題 1：資料庫選擇策略

## 如何為你的專案選擇合適的資料庫？

在開始一個新專案時，資料庫選擇是至關重要的決策。錯誤的選擇可能導致後續的效能瓶頸、維護困難甚至架構重構。以下提供系統性的選擇框架。

### 1. 資料模型與結構

首先評估你的資料型態。如果資料具有明確的關聯性（如訂單與客戶），關聯式資料庫如 PostgreSQL 是自然選擇。如果資料結構經常變動或包含大量巢狀文件，文件資料庫如 MongoDB 更為靈活。

```javascript
// 關聯式 vs 文件式資料模型比較
// 關聯式：需要預先定義綱要
const userSchema = `CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100),
  email VARCHAR(255) UNIQUE
)`

// 文件式：無需預先定義綱要
const userDoc = {
  name: 'Alice',
  email: 'alice@test.com',
  tags: ['developer', 'admin'],
  metadata: { lastLogin: new Date() }
}
```

### 2. 讀寫模式分析

分析應用的讀寫比例與模式。高讀取、低寫入的應用（如內容管理系統）適合搭配快取層如 Redis。高寫入負載的應用（如日誌系統）則需考慮寫入效能最佳化的資料庫。

### 3. 擴展需求

考量應用的未來成長。垂直擴展（升級硬體）對所有資料庫都適用，但水平擴展的需求會影響資料庫選擇。MongoDB 與 Cassandra 等 NoSQL 資料庫天生支援水平擴展，而 PostgreSQL 需要透過分割或讀取複本等方式實作。

### 4. 事務與一致性需求

如果應用需要強 ACID 保證（如金融系統），關聯式資料庫是最佳選擇。如果可以接受最終一致性，NoSQL 資料庫能提供更好的可用性與分割容忍性。

### 5. 團隊經驗與生態系統

團隊對特定資料庫的熟悉程度影響開發效率與維護成本。同時考量周邊工具、社群支援與文檔品質。

### 選擇流程總結

1. 定義資料模型與關聯性
2. 分析讀寫模式與效能需求
3. 評估擴展需求與預算限制
4. 確定事務與一致性要求
5. 考量團隊技術棧與學習曲線

### 常見錯誤與提醒

1. **過早優化**：不要為了未來可能的需求而選擇過度複雜的資料庫
2. **忽略運維成本**：某些資料庫雖然功能強大，但運維成本也相對較高
3. **混合使用**：必要時可組合多種資料庫，但需評估資料一致性的複雜度
4. **持續評估**：隨著應用成長，定期重新評估資料庫選擇是否仍然合適

### 常見的資料庫組合模式

現代的應用架構通常採用多種資料庫組合，充分發揮各自的優勢：

- **OLTP 資料庫**：PostgreSQL 或 MySQL 處理交易型工作負載
- **快取層**：Redis 或 Memcached 加速熱點資料讀取
- **搜尋引擎**：Elasticsearch 或 MeiliSearch 提供全文搜尋
- **分析資料庫**：ClickHouse 或 DuckDB 處理分析查詢
- **檔案儲存**：S3 或 MinIO 儲存使用者上傳檔案

### 實例：電商平台的資料庫選擇

以一個典型的電商平台為例：
- **商品目錄**：MongoDB（文件結構彈性，適合商品屬性多變）
- **訂單系統**：PostgreSQL（需要 ACID 事務保證）
- **購物車**：Redis（低延遲、暫時性資料）
- **搜尋功能**：Elasticsearch（全文搜尋與商品推薦）

透過這樣的組合，每個資料庫都發揮其最大優勢。
