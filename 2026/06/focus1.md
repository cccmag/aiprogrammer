# 資料管理的前世今生：從打孔卡到檔案系統（1890s-1960s）

## 資料處理的史前時代

在資料庫系統誕生之前，資料管理是一項極其原始的工作。程式設計師需要自行設計資料的儲存格式、管理 I/O 操作、處理資料的實體佈局。

### 打孔卡的時代

1890 年，美國人口普查局的 Herman Hollerith 發明了打孔卡（Punched Card）——這是人類歷史上第一個大規模資料處理工具。Hollerith 的公司後來演變成了 IBM。

```
打孔卡資料處理流程：
─────────────────────────

1. 原始資料（人口普查問卷）
         ↓
2. 打孔操作員在卡片上打孔
   ┌─────────────────────────────┐
   │ ██  ██  ██  ██  ██  ██     │
   │ ██  ██  ██  ██  ██  ██     │
   │ 性別  年齡  職業  地區       │
   └─────────────────────────────┘
         ↓
3. 打孔卡送入讀卡機
         ↓
4. 製表機進行計數和分類
         ↓
5. 產生統計報表
```

每張打孔卡可儲存 80 個字元——這個數字影響深遠，至今 Unix 終端機的預設寬度仍然是 80 字元。

### 磁帶與批次處理

1950 年代，磁帶（Magnetic Tape）成為主要的資料儲存媒體。磁帶的優勢是容量大、成本低；缺點是只能順序存取。

```
磁帶資料處理流程：
─────────────────────────

             ┌──────────┐
 交易資料 ──►│ 排序     │──► 排序後的交易
             └──────────┘
                      │
             ┌───────▼────────┐
 主檔案 ────►│ 合併/更新       │──► 新主檔案
             └────────────────┘
                      │
             ┌───────▼────────┐
             │ 報表產生        │──► 報表輸出
             └────────────────┘

特點：
- 只能順序處理
- 需要大量排序
- 批次處理（無法即時查詢）
- 使用「祖父-父親-兒子」備份策略
```

這種模式稱為「批次處理」（Batch Processing）——資料被分批載入、處理、輸出。查詢需要等待下一次批次執行，可能需要數小時甚至數天。

## 磁碟與隨機存取

### IBM 305 RAMAC

1956 年，IBM 發布了 305 RAMAC（Random Access Method of Accounting and Control）——世界上第一個磁碟儲存系統。RAMAC 使用了 50 個 24 英吋的碟片，總容量為 5MB。

```
IBM 305 RAMAC 規格：
─────────────────
容量：5 MB（當時是天文數字）
碟片：50 片 × 24 英吋
存取時間：約 600ms
體積：兩個大型冰箱
價格：每月 $3,200 美元租金
```

磁碟的革命性在於：**隨機存取**——不再需要從頭到尾讀取磁帶，可以直接定位到任何資料位置。

### CODASYL 與網路模型

1960 年代，資料管理領域出現了兩個重要的先驅系統：

**1. 階層式模型（Hierarchical Model）——IBM IMS**

1966 年，IBM 為 Apollo 計畫開發了 IMS（Information Management System）。IMS 使用樹狀結構組織資料：

```
                  ┌──────────┐
                  │  客戶     │
                  └────┬─────┘
                       │
         ┌─────────────┼─────────────┐
         │             │             │
    ┌────┴────┐   ┌────┴────┐   ┌────┴────┐
    │ 訂單 1  │   │ 訂單 2  │   │ 訂單 3  │
    └────┬────┘   └────┬────┘   └────┬────┘
         │             │             │
    ┌────┴────┐   ┌────┴────┐   ┌────┴────┐
    │產品 A   │   │產品 B   │   │產品 C   │
    └─────────┘   └─────────┘   └─────────┘
```

階層模型的問題在於：資料關係必須是嚴格的樹狀結構——「多對多」關係難以表達。

**2. 網路模型（Network Model）——CODASYL**

CODASYL（Conference on Data Systems Languages）提出了網路資料模型，使用「系」（Set）來表示記錄之間的關係，支援多對多關係：

```
    ┌──────────┐               ┌──────────┐
    │  學生 A  │               │  課程 X  │
    └────┬─────┘               └────┬─────┘
         │  ┌──────────────────┐    │
         └──┤    選課關係       ├────┘
            └──────────────────┘
    ┌──────────┐               ┌──────────┐
    │  學生 B  │               │  課程 Y  │
    └────┬─────┘               └────┬─────┘
         │  ┌──────────────────┐    │
         └──┤    選課關係       ├────┘
            └──────────────────┘
```

CODASYL DBTG（Data Base Task Group）在 1971 年發布了第一個資料庫標準規範。但無論是 IMS 還是 CODASYL，都有一個根本問題：**應用程式與資料的實體儲存方式緊密耦合**。

## 檔案系統的局限性

### 資料依賴問題

在資料庫系統出現之前，資料管理透過作業系統的檔案系統完成：

```c
// 1960 年代風格的資料存取
struct Employee {
    char name[30];
    int emp_id;
    float salary;
    char dept[20];
};

void read_employee(FILE* file, int id) {
    // 應用程式必須知道：
    // 1. 檔案的實體結構
    // 2. 記錄的格式
    // 3. 索引的實作方式
    // 4. 併發控制的邏輯
    
    fseek(file, id * sizeof(struct Employee), SEEK_SET);
    fread(&emp, sizeof(struct Employee), 1, file);
    
    // 查詢邏輯混合在應用程式碼中
    if (emp.salary > 50000) {
        printf("%s\n", emp.name);
    }
}
```

這種設計的缺點：

| 問題 | 說明 | 後果 |
|------|------|------|
| **資料重複** | 同一個資料存在多個檔案 | 資料不一致 |
| **資料依賴** | 程式碼依賴於檔案格式 | 修改格式需修改所有程式 |
| **無統一查詢** | 每種查詢都要寫不同的程式 | 開發效率低 |
| **無併發控制** | 多個程式同時寫入會損壞資料 | 資料損毀風險 |
| **無安全機制** | 檔案的存取權限粗糙 | 資料洩露風險 |
| **無災難恢復** | 當機時資料會遺失 | 資料可能永久丟失 |

### 為什麼需要資料庫？

```
應用程式視角：
─────────────────

沒有資料庫：
應用程式 ←→ 檔案系統（自行管理儲存格式、索引、併發、恢復）

有資料庫：
應用程式 ←→ 資料庫管理系統（DBMS）←→ 檔案系統（或直接儲存）

DBMS 提供的服務：
├── 資料抽象：應用程式只需關心「資料是什麼」，不需關心「怎麼存」
├── 資料獨立性：儲存格式改變不影響應用程式
├── 統一查詢：使用 SQL（結構化查詢語言）
├── 交易管理：ACID（原子性、一致性、隔離性、持久性）
├── 併發控制：多個使用者同時存取不會互相干擾
├── 安全管理：精細的存取權限控制
└── 災難恢復：當機後可恢復到一致狀態
```

## 結語

從 1890 年的打孔卡到 1960 年代的檔案系統，資料管理技術在七十年間經歷了從物理到邏輯的初步抽象。但直到 1970 年，關聯式資料庫的誕生才真正解決了資料管理中的核心問題。
