# 程式碼最佳化技術：從區域到全域最佳化（1970s-1990s）

## 最佳化的必要性

1970 年代，隨著高階語言的普及，編譯器生成的程式碼效能成為關鍵問題。程式設計師願意使用高階語言的前提是：編譯器生成的程式碼不能比手寫組合語言差太多。

### 為什麼需要最佳化？

```
┌─────────────────────────────────────────────────────┐
│              為什麼需要編譯器最佳化                    │
├─────────────────────────────────────────────────────┤
│                                                     │
│  原始碼（人類編寫）：                                │
│     for (int i = 0; i < 1000000; i++) {            │
│         sum += arr[i];                              │
│     }                                                │
│                                                     │
│  未最佳化（逐字翻譯）：                              │
│     LOAD  i, 0          ; 初始化 i                 │
│  L1: CMP   i, 1000000  ; 判斷條件                  │
│      JGE  L2           ; 如果條件不成立，跳出      │
│      LOAD  t1, arr     ; 載入陣列基底位址          │
│      MULT  t2, i, 4    ; 計算偏移量                │
│      ADD   t3, t1, t2  ; 計算元素位址              │
│      LOAD  t4, [t3]    ; 載入陣列元素              │
│      ADD   sum, sum, t4 ; 加到 sum                 │
│      ADD   i, i, 1     ; i++                       │
│      JMP   L1          ; 繼續迴圈                  │
│  L2:                     ; 結束                   │
│                                                     │
│  最佳化後：                                          │
│     ; 使用暫存器、減少記憶體存取、簡化迴圈          │
│     LOAD  r1, 0        ; sum in r1                │
│     LOAD  r2, 0        ; i in r2                  │
│     LOAD  r3, arr      ; base in r3               │
│  L1: CMP   r2, 1000000                             │
│      JGE  L2                                       │
│      ADD   r1, r1, [r3 + r2*4]   ; 直接記憶體存取 │
│      ADD   r2, r2, 1                               │
│      JMP   L1                                      │
│  L2:                                                │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 區域最佳化

區域最佳化（Local Optimization）在基本塊（Basic Block）內部進行——基本塊是沒有分支的連續指令序列。

### 常量摺疊（Constant Folding）

在編譯期計算常量表達式的值：

```
// 原始碼
x = 2 * 3 + 4;

// 常量摺疊後
x = 10;
```

### 常量傳播（Constant Propagation）

將已知的常量值傳播到使用它的位置：

```
// 原始碼
x = 5;
y = x + 3;
z = y * 2;

// 常量傳播與摺疊後
x = 5;
y = 8;    // 5 + 3
z = 16;   // 8 * 2
```

### 複製傳播（Copy Propagation）

消除不必要的複製操作：

```
// 原始碼
a = b;
c = a + d;

// 複製傳播後
c = b + d;  // 直接使用 b，消除 a
```

### 死碼消除（Dead Code Elimination）

移除永遠不會被使用的程式碼：

```
// 原始碼
x = 5;
y = x + 3;
x = 10;     // 這裡的 x 被重新賦值

// 如果 y 後續沒有被使用
x = 10;     // 前面的賦值都被消除
// y = x + 3 也被消除
```

### 代數化簡（Algebraic Simplification）

利用代數規則簡化運算：

```
// 原始碼
x = y + 0;
a = b * 1;
c = c * 0;

// 化簡後
x = y;
a = b;
c = 0;      // 任何數乘以 0 等於 0
```

## 全域最佳化

全域最佳化（Global Optimization）跨越基本塊的邊界，在整個函式或程式的範圍內進行。

### 資料流分析（Data Flow Analysis）

資料流分析是全域最佳化的核心技術。它在控制流圖（CFG）上求解資料流方程式：

```
// 控制流圖範例
         ┌─────┐
         │ B1  │  x = 5; y = 10;
         └──┬──┘
            │
         ┌──▼──┐
         │ B2  │  if (condition)
         └──┬──┘
        ╱        ╲
   ┌──▼──┐    ┌──▼──┐
   │ B3  │    │ B4  │  z = x + y;
   │ x=20│    │ y=15│
   └──┬──┘    └──┬──┘
        ╲        ╱
         ┌──▼──┐
         │ B5  │  w = x + y + z;
         └─────┘

// 到達定義（Reaching Definitions）分析
// 問題：在 B5 中，x 和 y 的值來自哪裡？
// 答案：x 可能來自 B1 (5) 或 B3 (20)
//       y 可能來自 B1 (10) 或 B4 (15)
```

常見的資料流分析問題：

| 分析類型 | 目的 | 應用 |
|---------|------|------|
| 到達定義 | 變數在哪裡被賦值 | 常量傳播、複製傳播 |
| 活躍變數 | 變數在哪些點被使用 | 暫存器分配 |
| 可用表達式 | 表達式是否已被計算 | 通用子表達式消除 |
| 向上使用的變數 | 哪些變數在下一次使用 | 死碼消除 |

### 通用子表達式消除（Global CSE）

在更大的範圍內消除重複計算：

```
// 原始碼
a = b * c + d;
e = b * c + f;

// 全域通用子表達式消除
t = b * c;
a = t + d;
e = t + f;
```

### 程式碼移動（Code Motion）

將不變的計算移出迴圈：

```
// 原始碼
for (int i = 0; i < n; i++) {
    x[i] = sqrt(2) * y[i] + z[i];
}

// 程式碼移動後
double tmp = sqrt(2);  // 移出迴圈！
for (int i = 0; i < n; i++) {
    x[i] = tmp * y[i] + z[i];
}
```

## 迴圈最佳化

迴圈是程式執行時間的熱點，因此迴圈最佳化是編譯器最重要的最佳化技術之一。

### 迴圈展開（Loop Unrolling）

減少迴圈控制開銷：

```
// 原始碼
for (int i = 0; i < 4; i++) {
    a[i] = b[i] + c[i];
}

// 展開後
a[0] = b[0] + c[0];
a[1] = b[1] + c[1];
a[2] = b[2] + c[2];
a[3] = b[3] + c[3];
```

### 迴圈歸納變數（Induction Variable Optimization）

用線性函式簡化歸納變數的計算：

```
// 原始碼
for (int i = 0; i < n; i++) {
    // 每次計算位址：arr + i * 4
    arr[i] = 0;
}

// 最佳化後：用另一個變數追蹤位址
int *ptr = arr;
for (int i = 0; i < n; i++) {
    *ptr = 0;
    ptr++;  // 指標遞增比乘法快得多
}
```

### 迴圈交換（Loop Interchange）

調整巢狀迴圈的順序以改善快取局部性：

```
// 原始碼（逐列存取）
for (int j = 0; j < N; j++) {
    for (int i = 0; i < M; i++) {
        a[i][j] = 0;
    }
}
// 問題：C 是 row-major，內層迴圈每次存取不同列
// 導致大量 cache miss！

// 交換後（逐行存取）
for (int i = 0; i < M; i++) {
    for (int j = 0; j < N; j++) {
        a[i][j] = 0;
    }
}
// 內層迴圈存取同一行，cache 局部性大幅提升
```

## RISC 與指令排程

### RISC 革命

1980 年代，David Patterson 和 John Hennessy 分別在 UC Berkeley 和 Stanford 推廣 RISC（Reduced Instruction Set Computer）架構。RISC 的核心思想：

- 精簡指令集：只有少數簡單指令
- 載入-儲存架構：只有 load/store 指令存取記憶體
- 固定指令長度：便於流水線處理
- 大量通用暫存器：減少記憶體存取

RISC 對編譯器的影響：

```
// CISC 風格（x86）
ADD [MEM_ADDR], 5  // 一條指令完成記憶體讀取、加法、儲存

// RISC 風格（ARM）
LOAD  r1, [MEM_ADDR]
ADD   r1, r1, 5
STORE [MEM_ADDR], r1
```

RISC 架構將複雜性從硬體移到軟體（編譯器），迫使編譯器承擔更多最佳化責任。

### 指令排程

在 RISC 和現代處理器中，指令的執行需要多個時鐘週期。指令排程（Instruction Scheduling）的目的是重排指令順序，減少流水線停頓：

```
// 原始指令順序（多個停頓）
LOAD  r1, [addr1]   // 停頓 3 週期（記憶體存取延遲）
ADD   r2, r1, r4   // 等待 r1
LOAD  r3, [addr2]   // 停頓 3 週期
SUB   r5, r3, r6   // 等待 r3

// 重新排程後（減少停頓）
LOAD  r1, [addr1]   // 開始第一次載入
LOAD  r3, [addr2]   // 立即開始第二次載入（無依賴）
ADD   r2, r1, r4   // 此時 r1 已到達
SUB   r5, r3, r6   // 此時 r3 已到達

// 收益：從 6+ 週期停頓減少到 0 週期停頓
```

### 軟體管線（Software Pipelining）

軟體管線是對迴圈的一種指令排程技術：

```
// 原始迴圈（每迭代都有起點和終點）
for i = 1 to n
    LOAD  r1, [a+i]     // 載入延遲 3 週期
    ADD   r1, r1, 1     // 計算
    STORE [a+i], r1     // 寫回
end

// 軟體管線化（重疊不同迭代的執行）
// 這樣每個週期都有一條指令在執行
LOAD  r1, [a+1]          // 預讀迭代 1
for i = 2 to n           // 主體
    LOAD  r2, [a+i]      // 預讀迭代 i
    ADD   r1, r1, 1      // 計算迭代 i-1
    STORE [a+i-1], r1    // 寫回迭代 i-1
    MOV   r1, r2         // 傳遞值
end
ADD   r1, r1, 1          // 處理最後一個
STORE [a+n], r1          // 寫回
```

## 最佳化的層級與取捨

不同層級的最佳化需要不同的編譯時間和效能提升：

| 最佳化層級 | 代表選項 | 編譯時間 | 執行效能 |
|-----------|---------|---------|---------|
| O0（無） | GCC -O0 | 最快 | 最慢 |
| O1（基本） | 區域最佳化 | 快 | 中等 |
| O2（中等） | 全域最佳化 | 中等 | 高效 |
| O3（進階） | 迴圈最佳化、內聯 | 慢 | 最高 |
| Os（大小） | 空間最佳化 | 中等 | 中等 |

### 最佳化的取捨

編譯器最佳化並非總是穩贏的：

1. **空間 vs 時間**：迴圈展開提升速度但增加程式碼大小
2. **編譯時間 vs 執行時間**：O3 編譯需要更長時間
3. **浮點精度 vs 速度**：某些重新關聯會改變浮點結果
4. **推測執行 vs 功耗**：推測執行可能浪費能源

## 結語

1970-1990 年代的最佳化技術演進，展示了編譯器從「如何正確翻譯」到「如何高效翻譯」的轉變。資料流分析、迴圈最佳化、指令排程等技術的成熟，使得編譯器生成的程式碼在很多情況下超越了手寫組合語言。

下一篇文章將探討 1990-2000 年代的即時編譯技術——當編譯器從「靜態」走向「動態」，從「編譯一次」走向「在執行時編譯」。
