# Tock OS：Rust 寫的安全嵌入式作業系統

## 前言

在物聯網裝置爆發的時代，嵌入式系統的安全性從未如此重要。傳統的即時作業系統（RTOS）如 FreeRTOS 雖然輕量高效，但它們大多以 C 語言撰寫，缺乏編譯期的記憶體安全保證。Tock OS 的誕生提供了一條截然不同的路徑：用 Rust 重寫整個作業系統核心，從根本上消除記憶體相關的安全漏洞。

## 架構與設計理念

### 分層架構

Tock OS 採用獨特的「核心 — 核心驅動 — 應用」三層架構：

```
┌─────────────────────────────────────┐
│           應用層 (Userspace)          │
│  C / Rust / Lua / Python 應用程式     │
├─────────────────────────────────────┤
│  ┌──────────┐  ┌──────┐  ┌──────┐  │
│  │  App 1   │  │App 2 │  │App 3 │  │
│  │ (Rust)   │  │ (C)  │  │(Lua) │  │
│  └────┬─────┘  └──┬───┘  └──┬───┘  │
├───────┴───────────┴──────────┴──────┤
│            核心驅動層 (Kernel)        │
│  ┌──────┐ ┌──────┐ ┌──────┐       │
│  │GPIO  │ │I2C   │ │UART  │  ...   │
│  │Driver│ │Driver│ │Driver│       │
│  └──────┘ └──────┘ └──────┘       │
├─────────────────────────────────────┤
│           硬體抽象層 (HAL)           │
│         Cortex-M / RISC-V           │
├─────────────────────────────────────┤
│              硬體 (MCU)              │
└─────────────────────────────────────┘
```

### 核心用 Rust，應用不用

Tock OS 最獨特的設計決策是：**核心必須用 Rust 撰寫**，但應用層可以選用任何語言。這是因為：

1. **核心是信任根（Root of Trust）**：一旦核心有記憶體漏洞，整個系統都會受影響
2. **應用層由 MPU 保護**：應用程式的記憶體錯誤會被硬體隔離，不會影響核心或其他應用
3. **降低進入門檻**：嵌入式開發者不需要精通 Rust 也能開發 Tock 應用

應用程式可以是 Rust、C、C++、甚至 Lua 等腳本語言，只要編譯成 Tock 的 ELF 格式即可載入執行。

### 無作業系統抽象化

Tock OS 不提供 POSIX 風格的抽象層。相反地，它直接暴露硬體中斷和事件，讓應用程式透過系統呼叫（syscall）與核心溝通。這降低了開銷，也讓系統行為更可預測，非常適合即時控制場景。

## 基於 Rust 型別系統的權限模型

Tock OS 最引人注目的特色是其基於 Rust 型別系統的權限模型，稱為 Capability（能力）。

### 為什麼需要 Capability

傳統嵌入式系統的權限檢查通常在執行期進行（例如用整數 ID 檢查某個 process 是否有權限存取 GPIO）。這種方式容易被繞過，也無法在編譯期保證安全。Tock 利用 Rust 的所有權（Ownership）和借用（Borrowing）規則，將權限變成編譯期的型別約束。

### Capability 的實現

在 Tock 的程式碼中，Capability 以空結構體（empty struct）搭配 unsafe trait 實現：

```rust
/// GPIO 的控制權限
pub struct GpioCapability {}

/// 標記哪些模組擁有特定能力
pub unsafe trait Capability {}
unsafe impl Capability for GpioCapability {}

/// 核心模組給予應用 GPIO 權限
pub fn grant_gpio_access(app: &mut Process) -> Result<GpioCapability, ()> {
    // 只有核心可以建立 Capability 實例
    if app.mpu_region_contains(MPU_REGION_GPIO) {
        Ok(GpioCapability {})
    } else {
        Err(())
    }
}

/// GPIO 操作需要 Capability 參數
pub fn set_pin_high(cap: &GpioCapability) {
    // 沒有正確的 Capability 就無法呼叫此函式
    // 這在編譯期就被保證
}
```

關鍵在於：**只有核心程式碼可以透過 `unsafe` 產生 `Capability` 實例**。應用程式無法繞過這個限制，因為它無法訪問核心的私有模組。這相當於將作業系統的權限檢查提升到了型別系統層級。

### 零成本抽象

Capability 在執行期完全沒有開銷。這些空結構體不佔用任何記憶體空間，編譯器會將它們優化為零成本標記。這對資源受限的嵌入式裝置至關重要。

## MPU 記憶體保護與應用隔離

### MPU 是硬體的防火牆

ARM Cortex-M 系列和 RISC-V 的嵌入式處理器通常配備 Memory Protection Unit（MPU）。MPU 可以將實體記憶體劃分為多個區域，並為每個區域設定存取權限（讀、寫、執行）。Tock OS 將 MPU 作為應用的硬體隔離邊界：

```
記憶體佈局（典型配置）：

0x20000000  ┌─────────────────────────┐
            │  核心堆疊與資料            │ ← 核心專屬，應用無法訪問
0x20004000  ├─────────────────────────┤
            │  App 1 棧空間             │
            │  App 1 堆空間             │
0x20008000  ├─────────────────────────┤ ← MPU 區域邊界
            │  App 2 棧空間             │
            │  App 2 堆空間             │
0x2000C000  ├─────────────────────────┤
            │  App 3 棧空間             │
            │  App 3 堆空間             │
0x20010000  └─────────────────────────┘
```

### 省略排程

Tock OS 不使用傳統的時間片輪詢（Round-Robin）排程。取而代之的是 **Event-Driven 協作式排程**：

1. 應用註冊它感興趣的事件（如 GPIO 中斷、計時器到期）
2. 事件發生時，核心喚醒對應的應用
3. 應用執行完一個操作後，自動讓出 CPU 回到等待狀態

這種設計減少了排程器的開銷，也簡化了並發控制。

### 核心與應用的訊息傳遞

Tock 使用 **Grant** 機制讓應用與核心安全共享記憶體。應用可以將一段記憶體「租借」給核心驅動程式使用，而核心保證不會越界存取。Grant 的型別安全由 Rust 的借用檢查器在編譯期確保。

## Tock OS 3.0 的新特性

2025 年第三季發布的 Tock OS 3.0 是該專案至今最大的一次升級，主要特性包括：

### 1. 多核心支援

3.0 版本開始支援對稱多處理（SMP），可以在多核心的 MCU（如 nRF5340 的雙核心架構）上執行。Tock 將不同核心分配給不同任務：一個核心專門處理即時控制，另一個處理網路協議棧。

### 2. 正交持久儲存（Orthogonal Persistence）

這是一項革命性的設計：應用程式的狀態在被中斷或暫停時可以自動序列化到 Flash。恢復執行時，應用完全不知道自己曾被暫停過。這對電池供電的 IoT 裝置極具價值，因為它們可以頻繁進入深度睡眠模式而不遺失狀態。

### 3. 使用者空間驅動程式（Userspace Drivers）

Tock 3.0 允許驅動程式以應用程式的形式運行在使用者空間。只有真正需要硬體特權操作的驅動（如中斷控制器、時鐘源）保留在核心。這大幅減小了可信計算基（TCB）的大小。

### 4. 改良的系統呼叫介面

新的系統呼叫 (`syscall`) 設計支援非同步操作，並引入 `Yield-Wait` 機制，讓應用可以有效地等待多個事件來源，而無需浪費 CPU 循環進行輪詢。

### 5. 工具鏈與開發體驗改善

Tock 3.0 引入了 `tockloader` 的完整重寫，支援更快的應用部署。同時也提供了 VSCode 擴展，包含即時的 MPU 配置可視化工具。

## Tock OS 與 FreeRTOS 的比較

FreeRTOS 是嵌入式領域最廣泛使用的 RTOS。以下是 Tock OS 與它的關鍵差異：

| 面向 | FreeRTOS | Tock OS |
|------|----------|---------|
| **核心語言** | C | Rust |
| **記憶體安全** | 依賴開發者經驗 | 編譯期保證（核心） |
| **應用語言** | C 為主 | 多語言（Rust、C、Lua 等） |
| **保護機制** | 無（或選擇性 MPU） | 強制 MPU 隔離 |
| **排程方式** | 搶佔式優先級 | 事件驅動協作式 |
| **權限模型** | 無（自由存取） | Capability 型別系統 |
| **最小 RAM** | ~1 KB | ~16 KB（含 MPU 配置） |
| **生態系統** | 極大，20+ 年歷史 | 成長中，但社群活躍 |
| **適用場景** | 資源極受限的 MCU | 中度以上資源、安全需求高 |

### 何時選擇 Tock OS

- 裝置需要運行來自不可信任第三方的應用
- 安全性是首要考量（如醫療、金融支付終端）
- 多語言開發團隊，部分開發者不熟悉 C
- 需要動態載入和卸載應用

### 何時選擇 FreeRTOS

- 極度資源受限的裝置（如 8 KB RAM 以下的 MCU）
- 對即時性要求極高（搶佔式排程更可控）
- 需要龐大的既有函式庫和驅動生態
- 團隊以 C 語言開發者為主

值得注意的是，這兩個系統並非完全不相容。Tock 社群中有專案嘗試在 FreeRTOS 上運行 Tock 應用二進位檔案，反過來也有 FreeRTOS 應用移植到 Tock 的成功案例。

## 實際應用場景

### 學術研究：嵌入式安全架構驗證

史丹佛大學和劍橋大學的研究團隊使用 Tock OS 來驗證嵌入式系統的安全模型。Rust 的型別系統使得研究人員能夠形式化地證明核心的安全性屬性，這在以 C 為基礎的 RTOS 上是極難實現的。

### 工業 IoT：感測器網路

在工業環境中，Tock OS 被部署在閘道器裝置上，同時運行多個感測器驅動、邊緣運算邏輯和雲端通訊協議。MPU 隔離確保一個感測器的驅動程式崩潰不會影響整個系統。

### 可穿戴裝置：智慧手錶

Tock OS 的事件驅動模型非常適合可穿戴裝置。應用可以註冊計時器和觸控事件後進入睡眠，只在需要時被喚醒。正交持久儲存功能讓應用狀態在裝置充電時自動保存。

### 安全性關鍵系統：門禁控制

門禁控制器需要同時運行通訊協議、卡片讀取器和鎖控制邏輯。Tock 的 Capability 模型確保卡片讀取驅動無法意外操作門鎖 GPIO，即使其中一個元件存在漏洞，整個系統仍然安全。

## 結語

Tock OS 不僅僅是另一個嵌入式作業系統，它是對嵌入式系統安全觀念的一次根本性重塑。透過將 Rust 語言的型別安全延伸到作業系統層級，Tock 證明了「安全不一定是效能的代價」。對於重視安全性和可靠性的嵌入式專案，Tock OS 是一個值得認真考慮的選擇。隨著 IoT 裝置數量持續增長，我們預期 Tock 類的安全架構將成為嵌入式領域的主流。
