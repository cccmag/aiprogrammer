# 多輪對話管理

## 從單輪到多輪

單輪對話系統處理的是獨立的問答對，每輪之間沒有關聯。然而，真實的對話具有複雜的結構：話題會變換、資訊會累積、意圖會演化。多輪對話管理正是為了解決這些問題而產生的。

## 對話結構

多輪對話可視為一個序列結構：

```
輪次 1：使用者 → 系統（問候）
輪次 2：使用者 → 系統（提出需求）
輪次 3：使用者 → 系統（補充資訊）
...
```

### 對話圖

複雜的對話可以用狀態圖表示，每個節點是一個對話狀態，邊代表使用者輸入觸發的轉移。

## 上下文建模

### 上下文窗口

最簡單的上下文建模方式是使用滑動窗口，只保留最近 K 輪的對話歷史：

```
context = h_{n-K} ⊕ h_{n-K+1} ⊕ ... ⊕ h_n
```

### 分層編碼

分層編碼方式分別編碼每輪對話，然後再編碼輪次序列：

```
turn_i = BERT(user_i, system_i)
context = BiLSTM(turn_1, turn_2, ..., turn_n)
```

### 記憶網路

記憶網路使用可讀寫的外部記憶體儲存對話歷史，系統可以根據需要查詢相關記憶。

## 對話流程控制

對話流程控制決定系統何時提問、何時回覆、何時轉移話題：

- **系統主動**：系統主導對話流程，逐步引導使用者提供所需資訊
- **使用者主動**：使用者控制對話方向，系統被動回應
- **混合主動**：系統和使用者共同控制對話

## 話題管理

在開放域對話中，話題管理是關鍵挑戰：

- **話題起始**：如何自然引入新話題
- **話題轉移**：如何流暢地從一個話題轉移到另一個
- **話題回溯**：如何回到之前討論過的話題
- **話題終止**：如何結束當前話題

## 實務架構

現代對話管理系統通常採用管線架構：

```
NLU → DST → Policy → NLG
```

每個組件負責不同的功能，形成一套完整的對話處理流程。
