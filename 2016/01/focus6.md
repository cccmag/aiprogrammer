# 6. 雲端架構設計原則

## 雲端架構 vs 傳統架構

傳統資料中心架構強調「預先規劃與過度配置」，因為採購硬體需要數週甚至數月，所以往往一次購買遠超過當前需求的容量。雲端時代強調「彈性與隨需擴展」，用多少買多少，不斷調整。

## 設計原則

### 1. 設計為失敗而生（Design for Failure）

在分散式系統中，元件故障是常態而非例外。雲端架構應該假設任何元件都可能失敗，並透過冗餘設計確保整體服務的可用性。具體做法包括：資料跨多個可用區域（Availability Zone）副本儲存、應用程式設計為無狀態以便負載平衡器分發請求、重要的資料定期備份並測試還原流程。

### 2. 鬆耦合（Loose Coupling）

將大型單體應用拆分為多個獨立的小服務，每個服務專注單一職責。服務之間透過定義良好的 API 介面溝通，而非直接相依。這種架構讓個別服務的更新、擴展、故障維護不會影響整體系統。微服務架構與 Serverless 函數是實作鬆耦合的兩種常見模式。

### 3. 水平擴展（Horizontal Scaling）

設計可橫向擴展的架構，而非依賴單一強大伺服器的垂直擴展。水平擴展的好處是：可隨負載動態調整容量、成本效益更高（多台小型機器通常比少數大型機器便宜）、不存在單一故障點。

### 4. 自動化（Automation）

雲端最大的優勢之一是可程式化（Infrastructure as Code）。使用 Terraform、CloudFormation、Ansible 等工具將基礎設施的設定寫成程式碼，確保環境可重現、易於審計、方便共同作業。自動化部署、自動化測試、自動化擴展都是成熟雲端架構的必備要素。

## 高可用性架構模式

**主動-被動（Active-Passive）**：正常運作時只有主站點處理流量，備援站點待命。故障時 DNS 切換到備援站點。適合資料庫等難以橫向擴展的元件。

**主動-主動（Active-Active）**：多個站點同時處理流量。負載平衡器或 DNS 輪詢分發請求到各站點。更好的資源利用率，但需要處理跨站點的資料同步問題。

**多可用區域（Multi-AZ）**：將資源部署在同一地區的多個可用區域內。AWS 的 AZ 是獨立的供電、網路基礎設施，一個 AZ 故障不會影響其他 AZ。可在不增加複雜度的情況下提升可用性。

## 雲端架構的常見錯誤

**錯誤一：將雲端當作傳統資料中心**：只是把應用程式遷移到雲端，但沒有重新設計架構以利用雲端的彈性。這樣只能獲得一小部分雲端的效益。

**錯誤二：缺乏監控與警示**：雲端環境的複雜度較高，更需要完善的監控、指標收集、異常警示機制。

**錯誤三：忽略成本優化**：沒有設定使用上限或警示，導致意外產生巨額帳單。
