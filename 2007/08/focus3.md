# 主題三：GPL 授權演進

## 自由軟體的法律保障

GNU General Public License (GPL) 是開源軟體世界中最具影響力的授權條款之一。從 1989 年 GPLv1 的誕生，到 1991 年 GPLv2 的演進，再到 2007 年 GPLv3 的討論，GPL 見證了自由軟體運動的發展。

## GPL 的起源

### 自由軟體運動

1983 年，Richard Stallman 發起了 GNU 計畫，旨在創建一個完全自由的作業系統：

```
GNU = GNU's Not Unix
GNU 計畫目標：
- 創造免費的自由軟體作業系統
- 確保軟體自由可以被保護
- 防止商業軟體佔用自由軟體的成果
```

### Copyleft 概念

GPL 的核心是「Copyleft」概念：

```
Copyleft 的定義：
- 你可以自由使用軟體
- 你可以自由修改軟體
- 你可以自由散布軟體
- 但你散布時必須保持同樣的自由

這是一種「傳染性」的授權
```

## GPL 版本演進

### GPLv1 (1989)

最初的 GPL 版本：

```markdown
主要條款：
1. 程式碼必須公開
2. 修改後的程式必須使用相同授權
3. 散布時必須保留版權聲明
```

### GPLv2 (1991)

成為 Linux 採用的授權：

```markdown
關鍵條款：
- 第 6 條：「不散布可讓你不再能散步程式的授權」被稱為「LIBRE」條款
- 這是專門為應對 AT&T/Unix 原始碼許可問題而設計的
```

### GPLv3 (2007 草案)

2007 年 GPLv3 草案引發了激烈討論：

```markdown
新增內容：
1. 專利授權條款
2. 與其他授權相容性（Apache License）
3. DRM 應對條款
4. 作業系統專利條款
```

### GPLv3 vs GPLv2 的主要差異

| 方面 | GPLv2 | GPLv3 |
|------|-------|-------|
| 專利保護 | 無明確條款 | 明確專利授權條款 |
| DRM | 無 | 回應 DMCA |
| 國際化 | 美國版權法 | 更國際化 |
| 與 Apache 2.0 相容 | 否 | 是 |

## GPL 的實際應用

### 常見的 GPL 軟體

```bash
# 作業系統
Linux Kernel          # GPLv2
GNU Coreutils          # GPLv3

# 開發工具
GCC                   # GPLv3
GDB                   # GPLv3
Make                  # GPLv3

# 辦公軟體
OpenOffice.org        # LGPL v3

# 資料庫
MySQL                 # GPLv2
PostgreSQL            # PostgreSQL License
```

### GPL 的義務

```markdown
散布 GPL 軟體時，你必須：

1. 保留版權聲明和授權聲明
2. 提供完整原始碼或書面承諾
3. 明確說明程式的修改之處
4. 說明如何取得原始碼
5. 附上 GPL 授權全文
```

### LGPL (Lesser GPL)

對於函式庫的寬鬆版本：

```markdown
LGPL 允許：
- 靜態連結到專有軟體
- 動態連結時必須允許替換函式庫

LGPL 應用場景：
- Gtk+ (用於 Qt 間接 GPL 相容)
- glibc
- ICU
```

## GPL 與商業軟體

### 商業軟體使用 GPL 軟體

```markdown
問題：我的商業軟體可以使用 GPL 程式碼嗎？

答案：**不行**

原因：GPL 的「傳染性」要求衍生作品必須也是 GPL

解決方案：
1. 購買商業授權（如 MySQL 的商業授權）
2. 使用非 GPL 的替代方案
3. 保持你的程式碼與 GPL 程式碼分離（通过 IPC，而非静态链接）
```

### GPL 的「母公司」問題

2007 年，一個重要的法律問題是 GPL 的「母公司」條款：

```markdown
問題：
如果公司 A 擁有公司 B 的多數股份，
公司 B 發布的 GPL 軟體是否會「傳染」公司 A 的專有軟體？

社群立場：應該不會，因為公司與子公司是獨立法人

律師立場：取決於具體情況
```

## BSD/MIT 與 GPL 的對比

### 寬鬆授權

BSD 和 MIT 授權比 GPL 更加寬鬆：

```markdown
BSD/MIT 允許：
✓ 使用原始碼
✓ 修改原始碼
✓ 散布原始碼
✓ 散布二進制
✓ 用於商業產品
✓ 專有軟體可以使用

BSD/MIT 不要求：
✗ 必須公開修改後的原始碼
✗ 必須使用相同授權
```

### 比較表

| 方面 | GPL | LGPL | BSD/MIT |
|------|-----|------|---------|
| 可用於商業 | ✓ | ✓ | ✓ |
| 可修改後閉源 | ✗ | ✗ | ✓ |
| 必須開源 | ✓ | 僅函式庫 | ✗ |
| 必須相同授權 | ✓ | ✗ | ✗ |

## GPL 社群影響

### 授權遵守

```markdown
GPL 的執行：
- 依賴社群自覺
- 如果被發現違反，可能被起訴
- FSF 會追蹤違規者

常見違反：
1. 使用 GPL 程式碼但不散布原始碼
2. 移除版權聲明
3. 不提供修改說明
```

### 授權選擇

```markdown
選擇 GPL 的原因：
1. 保護軟體自由
2. 確保衍生作品也保持開源
3. 社區支持

選擇 BSD/MIT 的原因：
1. 允許商業使用
2. 最大化合作品可能性
3. 吸引企業採用
```

## 結語

GPL 授權是自由軟體運動的重要法律保障。從 GPLv1 到 GPLv3 的演進，反映了自由軟體社群對新技術挑戰的回應。理解 GPL 的精神和條款，對於任何參與開源軟體開發的人來說都是必修課題。
