## 濃縮回顧

### NoSQL 的起源

```
NoSQL 運動時間線：

2004年：Google Bigtable 論文
     ↓
2006年：Amazon Dynamo 論文
     ↓
2008年：Cassandra 開源
     ↓
2009年：MongoDB 1.0 發布
     ↓
2010年：NoSQL 這個詞廣泛流傳
```

### 為什麼需要 NoSQL？

```
傳統 SQL 資料庫的挑戰：

1. 規模化困難
   - 垂直擴展昂貴
   - 水平擴展複雜

2. 彈性不足
   - 固定 Schema
   - 難以適應快速變化

3. 複雜查詢瓶頸
   - JOIN 成本高
   - 全文檢索支援有限

NoSQL 的解決方案：
+ 水平擴展簡單
+ 彈性 Schema
+ 針對特定場景優化
```

### 文件資料庫的優勢

```
文件 vs 關聯式：

關聯式：
Users 表 → Orders 表（透過外鍵）
使用 JOIN 組合資料

文件：
{
  "user_id": 1,
  "name": "張三",
  "orders": [
    {"id": 101, "total": 500},
    {"id": 102, "total": 300}
  ]
}

優點：
- 單一文件包含所有相關資料
- 無需昂貴的 JOIN
- 讀取效率高
```

## 結論與展望

NoSQL 運動代表了對傳統關聯式資料庫的補充而非取代。2009 年，MongoDB 1.0 的發布標誌著文件資料庫進入實用階段。

展望未來：

1. **多模型資料庫**：結合關聯式和文件模型的優勢
2. **更強的一致性模型**：吸取教訓，提供更靈活的一致性控制
3. **標準化努力**：GraphQL 和其他查詢語言的出現
4. **雲端原生**：專為雲端設計的資料庫
