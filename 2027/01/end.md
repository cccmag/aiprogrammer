### 本期回顧

本期聚焦於 Rust 在機器學習推論中的角色——不是訓練，而是部署。

**訓練用 Python，部署用 Rust**——這是本期反覆出現的主旋律。Python 在 ML 訓練領域的統治地位短期內不會改變，但部署場景的需求截然不同：

1. **冷啟動時間**：Rust 的二進位在微秒級啟動，Python 需要數百毫秒
2. **資源消耗**：Rust 的精確記憶體控制使其可以在數百 KB 的 MCU 上運行
3. **交叉編譯**：從 x86 伺服器到 ARM 邊緣裝置，Rust 只需切換 target
4. **併發推論**：無 GC、無 GIL 的架構讓高吞吐推論伺服器更加高效

**Candle、Burn、tract**——三大框架各有定位：

- **Candle**：最輕量，無外部依賴，適合嵌入式場景和 Hugging Face 模型
- **Burn**：最靈活，可組合後端設計，適合需要 GPU 加速的跨平台部署
- **tract**：最標準，原生 ONNX 支援，適合需要模型互通性的企業場景

**迷你 ML 推論引擎（mini-ml）** 是本月最具教育意義的程式專案。它用約 200 行 Rust 程式碼展示了張量運算、線性層、激勵函式與前向傳播的核心機制。理解 mini-ml 之後，Candle/Burn/tract 的內部運作就不再有黑盒子。

### Rust ML 生態的現狀與挑戰

**現狀**：Rust ML 生態已覆蓋大部分推論場景。從伺服器端的模型服務（Model Serving）到邊緣裝置的即時推論，Rust 都已有了成熟的框架選擇。Hugging Face 對 Candle 的官方支援、tract 在 Android 中的整合，都說明了 Rust ML 的生態影響力。

**挑戰**：

1. **訓練支援仍有限**：Burn 支援訓練，但生態遠不及 PyTorch。Rust ML 的主要定位仍是部署
2. **GPU 生態依賴**：雖然 WGPU 後端已經成熟，但 CUDA GPU 的支援仍需仰賴外部 C 函式庫
3. **社群規模**：相比 Python ML 社群，Rust ML 的開發者基數仍然較小，第三方模型的 Rust 移植進度不一
4. **工具鏈整合**：模型壓縮、量化校準、佈署測試的工具鏈仍在發展中

### Rust ML 學習路徑

給想要進入 Rust ML 領域的讀者：

1. **基礎**：先掌握 Rust 語言本身（所有權、trait、泛型），再學習 mini-ml 理解張量運算核心
2. **框架入門**：從 Candle 開始（最簡單的 API），然後嘗試 tract（ONNX 互通），最後探索 Burn（進階後端設計）
3. **實戰專案**：嘗試從 PyTorch 訓練一個簡單模型，匯出 ONNX，用 tract 在 Rust 中執行推論
4. **邊緣部署**：在 Raspberry Pi 上部署 Rust ML 推論管線，加入感測器資料處理
5. **量化與最佳化**：學習模型量化技術，將 FP32 模型壓縮為 INT8 並在邊緣裝置上運行
6. **生產部署**：設計低延遲的非同步推論伺服器，整合 Prometheus 監控和模型版本管理
