# 系統設計思維

## 從需求到架構

系統設計不僅僅是選擇技術棧，更是一種思考方式。它要求我們在混亂的需求中找出核心問題，在有限的資源下做出最佳取捨。

### 需求分析的四個層次

1. **功能需求**：系統要做什麼？例如「用戶可以發送訊息」
2. **非功能需求**：系統要達到什麼品質？例如「訊息延遲小於 100ms」
3. **約束條件**：有哪些限制？例如「僅有 3 人團隊，預算有限」
4. **規模估算**：系統要支撐多少流量？例如「月活用戶 100 萬」

## 規模估算

### 流量估算技巧

以 Twitter 為例，估算每日推文量：

```
月活用戶：3 億
每日活躍：1.5 億（50%）
每人每天發推：2 條
每日推文總量：3 億條
每秒推文量（QPS）：3億 / 86400 ≈ 3500
高峰 QPS：3500 × 3 = 10500
```

### 資料儲存估算

```
每日推文量：3 億條
每條推文約：1 KB
每日新增儲存：3億 × 1KB ≈ 286 GB
每月新增儲存：286 GB × 30 ≈ 8.4 TB
五年總儲存：8.4 TB × 60 ≈ 504 TB
```

這樣的估算幫助我們選擇合適的儲存方案——504 TB 顯然需要分散式儲存系統。

## Trade-off 的藝術

系統設計中沒有完美的方案，每一個選擇都有代價。

### 常見取捨

**效能 vs 一致性**

```
選擇高一致性
  優點：資料準確，開發簡單
  代價：寫入延遲高，吞吐量受限

選擇最終一致性
  優點：高吞吐量，低延遲
  代價：短時間內資料可能不一致
```

**延遲 vs 吞吐量**

```
選擇低延遲
  優點：用戶體驗好
  代價：需要更多資源，成本高

選擇高吞吐量
  優點：節省成本
  代價：單一請求延遲較高
```

**開發速度 vs 系統品質**

```
快速迭代（MVP）
  優點：快速驗證市場
  代價：技術債累積，後期重構成本高

穩健設計
  優點：可擴展性強
  代價：前期開發週期長
```

## CAP 定理

分散式系統中，Consistency（一致性）、Availability（可用性）、Partition Tolerance（分區容錯性）三者不可兼得。

```
┌─────────────────────────────────┐
│            CAP 定理              │
├─────────────────────────────────┤
│                                 │
│       CP 系統                    │
│   （犧牲可用性）                  │
│   例：ZooKeeper, etcd           │
│                                 │
│       AP 系統                    │
│   （犧牲一致性）                  │
│   例：Cassandra, DynamoDB       │
│                                 │
│       CA 系統                    │
│   （網路故障時不可用）            │
│   例：單機關聯式資料庫            │
│                                 │
└─────────────────────────────────┘
```

### 實務中的 CAP

在真實系統中，即使在 CP 模式下，我們也可以通過以下手段緩解可用性問題：

- **Quorum 機制**：不必等待所有節點回應
- **Hinted Handoff**：暫時不可用的節點後續補償
- **Read Repair**：讀取時修復不一致的資料

## BASE 原則

與 ACID 相對，BASE 是 NoSQL 系統常用的設計哲學：

```
B - Basically Available（基本可用）
  系統保證大部分時間可用，可能部分功能受損

A - Soft State（軟狀態）
  系統的狀態可能隨時間變化，不需持續一致

E - Eventually Consistent（最終一致性）
  經過足夠時間後，所有副本最終會達成一致
```

### 何時使用 BASE？

適合場景：
- 社群媒體的時間線（短暫不一致可以接受）
- 電商平台的庫存顯示（不要求即時精確）
- 內容推薦系統（延遲更新無妨）

不適合場景：
- 銀行轉帳（需要強一致性）
- 票務系統（防止超賣）

## 系統設計的思考框架

### 步驟一：理解需求
- 問清楚功能需求和非功能需求
- 明確使用者規模和流量模式

### 步驟二：高層次設計
- 畫出系統的整體架構圖
- 標示主要元件和資料流

### 步驟三：深度設計
- 針對每個元件進行詳細設計
- 討論 API 設計、資料模型、儲存方案

### 步驟四：識別瓶頸
- 哪些部分是潛在的單點故障？
- 哪些元件的吞吐量可能不足？

### 步驟五：迭代優化
- 根據瓶頸調整設計
- 評估改進方案的取捨

## 實戰案例：設計 URL 縮短服務

**需求**：建立一個類似 tinyurl 的服務

**估算**：
- 每日新增 1 萬長 URL
- 每年約 365 萬條
- 總 URL 數約 1 億（十年）
- 讀寫比約 100:1

**設計決策**：
1. 用 Base62 編碼產生短網址（62^7 ≈ 3.5 兆組合）
2. 用 Redis 做讀取快取（讀多寫少）
3. 用關聯式資料庫做主儲存
4. 寫入時先寫資料庫再更新快取

**關鍵取捨**：
- 短網址長度 vs 容量：7 字元足夠到 3.5 兆
- 隨機字串 vs 有序 ID：隨機更安全，有序更快
