# 系統設計面試思維

## 破解系統設計面試

## 系統設計面試的本質

系統設計面試不是考你「知道多少技術」，而是考察你**如何在開放式問題中做架構決策**。面試官想知道：

1. 你有沒有系統化的思考框架？
2. 你是否能辨識關鍵取捨？
3. 你的溝通能力是否清晰？

---

## 四步答題框架

### 第一步：釐清需求（3-5 分鐘）

在動手設計之前，先問清楚問題。

```
面試官：「請設計一個 URL 縮短服務」
```

你應該問：

**功能需求**
- 支援哪些操作？建立短網址、訪問短網址
- 需要自訂別名嗎？
- 需要分析統計嗎（點擊次數、來源）？

**非功能需求**
- 每天的短網址產生量？
- 整體連結數量？
- 讀寫比例？
- 延遲要求？

**規模估算**
假設每天產生 100 萬新短網址，年總量約 3.6 億，讀寫比約 100:1。

### 第二步：高層次設計（5-10 分鐘）

畫出整體架構圖，不需要細節，而是展示所有主要元件。

```
Client ──→ Web Server ──→ Application ──→ Database
                                      └──→ Cache
```

解釋每個元件的角色：
- Web Server：接收 HTTP 請求
- Application：業務邏輯
- Database：持久化儲存
- Cache：加速讀取

### 第三步：深度設計（10-15 分鐘）

針對關鍵元件深入討論。

**如何產生短網址？**

```
方案 A：Hash + 碰撞處理
  MD5(long_url) → 取前 7 字元 → Base62 編碼
  碰撞時加鹽後重試

方案 B：分散式 ID 產生器
  資料庫自增 ID → Base62 編碼
  使用 Redis 或 Snowflake 產生 ID
```

為什麼選 B？因為 Hash 方案需要處理碰撞，且 Hash 值不可逆。

### 第四步：總結與改進（2-3 分鐘）

指出設計的不足之處和改進方向。

```
瓶頸分析：
- 寫入瓶頸：資料庫單點
- 讀取瓶頸：可透過快取解決

改進方向：
- 水平擴展資料庫
- 使用 CDN 快取熱門連結
- 加入資料庫分片
```

## 常見陷阱與應對

### 陷阱一：過早深入細節

```
❌ 「我用 Redis Cluster 做 Session 管理， 
     它有 16384 個 Hash Slot...」

✅ 先說整體架構，再逐步深入
```

**正確做法**：先給高層次架構圖，等面試官追問時再深入。

### 陷阱二：忽略取捨

```
❌ 「我用 NoSQL，因為它快」

✅ 「我用 Cassandra，因為寫入量很大，
    雖然犧牲了強一致性，但換來了高吞吐量」
```

### 陷阱三：只說優點不說缺點

```
❌ 「微服務是完美的解決方案」

✅ 「微服務可以獨立部署，
    但增加了分散式事務和除錯的複雜性」
```

## 評分標準

```
是否涵蓋功能需求：       20%
規模估算是否合理：       10%
架構是否清晰可擴展：     30%
是否討論取捨和瓶頸：     20%
溝通表達是否流暢：       20%
```

## 模擬答題範例

### 設計 URL 縮短服務

**需求釐清**
- 每天 100 萬新連結
- 每年約 3.6 億
- 10 年約 36 億
- 讀寫比 100:1

**高層次設計**
```
Client → Load Balancer → API Server → Cache (Redis)
                                       → DB (PostgreSQL)
```

**關鍵決策**
1. 短網址長度：7 字元（62^7 ≈ 3.5 兆）
2. 編碼方式：Base62
3. ID 產生：資料庫 auto increment → Base62
4. 快取策略：Cache-Aside，TTL 30 天
5. 301 重定向（永久）vs 302 重定向（臨時）

**取捨討論**
- 讀寫分離：主資料庫寫入，從資料庫讀取
- 快取更新：先更新資料庫，再刪除快取
- 熱點問題：熱門連結可自動延長 TTL
