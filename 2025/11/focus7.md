# 微服務 vs 單體架構

## 架構選擇的藝術

## 單體架構

### 什麼是單體架構？

單體架構將所有功能打包在一個應用程式中，共用同一個進程和資料庫。

```
┌─────────────────────────────────────┐
│          Monolith Application        │
│                                      │
│  ┌─────────┐  ┌──────────┐          │
│  │  用戶模組 │  │  訂單模組 │          │
│  └─────────┘  └──────────┘          │
│  ┌─────────┐  ┌──────────┐          │
│  │  商品模組 │  │  支付模組 │          │
│  └─────────┘  └──────────┘          │
│                                      │
├─────────────────────────────────────┤
│          共享資料庫                    │
└─────────────────────────────────────┘
```

### 單體架構的優點

**開發初期效率高**
- 單一程式碼庫，不需要跨服務協調
- 本地開發環境簡單
- 一次部署即包含所有功能

**效能**
- 模組間呼叫是同一進程內的函式呼叫（微秒級）
- 沒有網路開銷
- 事務一致性容易保證

**除錯與監控**
- 單一日誌檔案即可追蹤完整請求
- 簡單的應用程式效能監控
- 不需要分散式追蹤工具

### 單體架構的缺點

**隨著規模擴大，問題浮現：**

1. **部署耦合**：修改一行程式碼需要重新部署整個應用
2. **擴展粒度粗**：無法只擴展瓶頸模組（如支付），必須擴展整體
3. **技術棧綁定**：整個應用使用同一語言和框架
4. **團隊協作困難**：多人修改同一程式碼庫，合併衝突頻繁
5. **啟動時間長**：大型單體應用啟動可能需要數分鐘

## 微服務架構

### 什麼是微服務？

微服務將應用拆分為多個獨立部署的小型服務，每個服務負責單一業務領域。

```
┌─────────────────────────────────────────┐
│            API Gateway                  │
└─────┬──────┬──────┬──────┬──────┬──────┘
      │      │      │      │      │
  ┌───▼┐ ┌──▼──┐ ┌▼───┐ ┌▼───┐ ┌▼───┐
  │用戶│ │訂單│ │商品│ │支付│ │通知│
  │服務│ │服務│ │服務│ │服務│ │服務│
  └───┘ └───┘ └───┘ └───┘ └───┘
  │     │     │     │     │
  DB    DB    DB    DB    DB
  (各服務獨立資料庫)
```

### 微服務的優點

**獨立部署**：每個服務可以獨立開發、測試、部署、擴展

**技術多樣性**：不同服務可以使用不同的技術棧

```
用戶服務：Python + PostgreSQL
訂單服務：Go + MySQL
商品服務：Java + Elasticsearch
搜尋服務：Node.js + Redis
```

**故障隔離**：一個服務故障不會影響其他服務

**團隊自主**：小團隊可以全權負責自己的服務

### 微服務的缺點

**分散式系統的複雜性**
- 服務間通訊需要網路呼叫（毫秒級）
- 需要處理部分失敗
- 資料一致性更困難（最終一致性）

**運維成本高**
- 需要容器化、編排平台（K8s）
- 需要服務發現、負載平衡
- 需要分散式日誌和追蹤

**開發初期成本高**
- 需要建立 CI/CD 管道
- 需要基礎設施自動化
- 團隊需要學習分散式系統知識

## 何時選擇單體？

| 場景 | 建議 |
|------|------|
| 團隊 1-5 人 | 單體 |
| 產品未驗證市場 | 單體 |
| 業務邏輯簡單 | 單體 |
| 部署頻率低（每月幾次） | 單體 |
| 對延遲極度敏感 | 單體 |

## 何時遷移到微服務？

| 場景 | 建議 |
|------|------|
| 團隊超過 10 人 | 考慮拆分 |
| 多個功能需要獨立擴展 | 微服務 |
| 部署頻率高（每天多次） | 微服務 |
| 部分功能需要不同技術棧 | 微服務 |
| 單體應用已變得難以維護 | 考慮重構 |

## 模組化單體：折衷方案

在完全單體和微服務之間，存在一個中間地帶。

```
┌─────────────────────────────────────┐
│            Module Monolith           │
│                                      │
│  ┌─────────┐      ┌──────────┐      │
│  │ Module A │      │ Module B │      │
│  │ (package)│      │ (package)│      │
│  └─────────┘      └──────────┘      │
│  ┌─────────┐      ┌──────────┐      │
│  │ Module C │      │ Module D │      │
│  │ (package)│      │ (package)│      │
│  └─────────┘      └──────────┘      │
│                                      │
│  # 模組間透過定義良好的介面通訊        │
│  # 但仍在同一進程中                   │
└─────────────────────────────────────┘
```

**優點**：保留單體的簡單性，同時強制關注點分離
**遷移路徑**：當模組化單體仍不足時，可以逐步將模組拆分為獨立微服務

## 遷移策略：絞殺者模式（Strangler Fig）

逐步將單體功能遷移到微服務，而不是一次性重寫。

```
階段 1：單體
  [所有功能都在單體中]

階段 2：開始拆分
  [單體 → 用戶功能]  +  原始單體（其他功能）
  
階段 3：持續拆分
  [單體 → 用戶功能]  [單體 → 訂單功能]  +  原始單體
  
階段 4：完成
  [用戶服務]  [訂單服務]  [商品服務]  [支付服務]
```

### 絞殺者模式的好處

- **風險可控**：逐步遷移，每次只改一個功能
- **持續交付**：不阻塞新功能開發
- **容易回滾**：發現問題時可以保留單體版本
