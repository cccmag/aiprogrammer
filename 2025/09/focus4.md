# 測試驅動開發 TDD

## 紅-綠-重構循環的藝術

### 前言

測試驅動開發（Test-Driven Development，TDD）是 Kent Beck 在 1990 年代提出的一種開發方法論。與傳統的「先寫程式碼、再寫測試」不同，TDD 要求開發者**先寫測試，再寫程式碼**。這個看似簡單的倒轉，卻深刻地改變了軟體開發的方式。

### TDD 的核心循環：紅-綠-重構

TDD 遵循一個稱為「紅-綠-重構」（Red-Green-Refactor）的三步驟循環：

```
[紅] 寫一個會失敗的測試
     ↓
[綠] 寫最簡單的程式碼讓測試通過
     ↓
[重構] 在不改變行為的前提下改善程式碼
     ↓
     (回到第一步，加入下一個測試)
```

**第一步：紅（Red）**

先撰寫一個測試，描述你期望的行為。因為還沒有實作，測試會失敗。

```python
# 先寫測試
def test_add():
    calc = Calculator()
    assert calc.add(2, 3) == 5  # Calculator 還不存在
```

**第二步：綠（Green）**

寫最簡單的程式碼讓測試通過。不要過度設計，不要提前實作未測試的功能。

```python
# 寫剛剛好夠用的程式碼
class Calculator:
    def add(self, a, b):
        return a + b
```

**第三步：重構（Refactor）**

測試通過後，你可以安全地重構程式碼——改進命名、提取方法、優化結構——因為測試會確保你不會破壞既有功能。

```python
# 重構：加入型別提示和文件字串
class Calculator:
    def add(self, a: float, b: float) -> float:
        """回傳兩個數字的和"""
        return a + b
```

### 實際案例：開發 Fibonacci 函數

讓我們用 TDD 開發一個 Fibonacci 函數。

**第一輪：處理 n=0**

```python
# 測試
def test_fib_zero():
    assert fib(0) == 0

# 實作
def fib(n):
    return 0
```

**第二輪：處理 n=1**

```python
def test_fib_one():
    assert fib(1) == 1

def fib(n):
    if n == 0:
        return 0
    return 1
```

**第三輪：處理 n=2**

```python
def test_fib_two():
    assert fib(2) == 1

def fib(n):
    if n <= 1:
        return n
    return 1
```

**第四輪：推廣到一般情況**

```python
def test_fib_ten():
    assert fib(10) == 55

def fib(n):
    if n <= 1:
        return n
    return fib(n-1) + fib(n-2)
```

每一步，測試都確保之前的行為不受影響。這正是 TDD 的力量——你永遠不會回歸（regression）。

### TDD 的優點

**更好的設計**

當你先寫測試，你被迫思考「這個程式碼的介面應該是什麼樣子的？」而不是「這個程式碼內部應該怎麼實作？」。這種外部視角通常會導致更好的 API 設計——更簡潔、更一致、更容易使用。

**更高的覆蓋率**

TDD 天然地產生高覆蓋率。因為測試是在實作之前寫的，不太可能出現「忘記測試某個分支」的情況。

**更低的除錯成本**

TDD 的測試粒度通常很小——每個測試只驗證一個行為。當測試失敗時，你可以立即定位到問題所在，不需要花費大量時間在除錯上。

**安全的重構**

有了完整的測試套件，你可以放心地重構程式碼。測試就像安全網——你可以大膽嘗試不同的實作方式，只要測試通過，你就知道功能是正確的。

### TDD 的挑戰

**學習曲線**

TDD 違反了大多數開發者的直覺。剛開始時，你可能會覺得「先寫測試」很彆扭，覺得自己在繞遠路。這需要一段時間的練習才能習慣。

**不適用於所有場景**

TDD 最適合邏輯密集的領域（如業務規則、演算法、資料處理）。對於 UI 開發、探索性原型、或者需求極不明確的專案，TDD 的效益可能較低。

**需要紀律**

TDD 是一個需要嚴格遵守的紀律。最常見的失敗模式是：「我先寫一點程式碼，待會再補測試」——然後就再也沒有補寫測試。

### 什麼時候該使用 TDD？

TDD 特別適合以下場景：
- **複雜的業務邏輯**：財務計算、訂單處理、庫存管理
- **需要長期維護的專案**：測試是對未來開發者的承諾
- **演算法實作**：清晰的輸入/輸出關係最適合 TDD
- **修 Bug**：先寫一個會失敗的測試來重現 Bug，然後修復它

### TDD 常見的誤解

**誤解 1：TDD 會讓開發變慢**

事實是：TDD 在短期內確實會讓初始開發速度變慢，但在長期來看，由於減少了除錯時間和回歸 Bug，總開發時間反而是減少的。

**誤解 2：TDD 需要完美的測試覆蓋率**

TDD 的核心不是 100% 覆蓋率，而是**測試驅動設計**。有些邊界條件可能不值得測試，你應該用工程判斷來決定測試的投資回報率。

**誤解 3：TDD 只能在綠地專案中使用**

TDD 同樣適用於既有程式碼庫。當你需要在既有系統中加入新功能時，先為新功能寫測試，然後逐步為相關的既有程式碼補上測試。

### 小結

TDD 不僅是一種測試技術，更是一種設計方法論。它迫使開發者從使用者的角度思考 API 設計，從需求的角度思考實作策略。紅-綠-重構的循環不僅產出高品質的程式碼，更培養了開發者對程式碼品質的敏感度。
