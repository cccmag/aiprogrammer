# 手動測試 vs 自動化測試

## 前言

在軟體開發中，「測試」涵蓋了從開發者在 IDE 中隨手執行的 `print` 除錯，到 CI/CD 管線中數百個自動化測試案例的執行。這些測試方式大致可分為兩大類：手動測試和自動化測試。了解兩者的優缺點和適用場景，是制定有效測試策略的第一步。

## 手動測試

手動測試是由真人測試人員操作應用程式，檢查功能是否正常。這是最傳統的測試方式。

### 優點

**探索性測試**：手動測試人員可以發現自動化腳本會遺漏的問題——奇怪的邊界行為、UI 顯示異常、使用者體驗不佳等。人類的直覺和創造力是自動化腳本無法替代的。

**適合一次性測試**：對於只需要驗證一次的場景（如上線前的煙霧測試），撰寫自動化腳本的投資報酬率很低。

**適合 UI/UX 驗證**：佈局是否美觀、動畫是否流暢、顏色是否協調——這些主觀感受只能由人類判斷。

### 缺點

**耗時且昂貴**：每次回歸測試都需要手動執行所有步驟，隨著系統規模增長，手動測試的時間成本呈線性成長。

**容易出錯**：人類會疲勞、分心、遺漏步驟。同一個測試案例，不同的人、不同的時間執行可能得到不同的結果。

**不可重複**：手動測試的執行過程難以精確重現，這讓 Bug 的定位和修復變得困難。

## 自動化測試

自動化測試是透過腳本和工具自動執行測試案例的方式。從單元測試到 E2E 測試，都可以實現自動化。

### 優點

**快速**：自動化測試可以在數分鐘內執行數千個測試案例，這是手動測試無法比擬的。

**可靠**：只要測試腳本正確，自動化測試每次執行都會得到相同的結果。

**可重複**：你可以在任何時間、任何環境執行相同的測試，獲得一致的結果。

**持續整合**：自動化測試是 CI/CD 管線的基礎——每次程式碼提交都可以自動觸發測試執行。

### 缺點

**初始成本高**：撰寫自動化測試腳本需要時間和技術投入。

**維護成本**：當程式碼變更時，測試腳本也需要更新。UI 測試尤其脆弱——按鈕位置的微小變化可能導致整個測試失敗。

**無法發現意外問題**：自動化測試只驗證你寫了斷言的那些行為，無法像人類一樣發現未預期的問題。

## 適用場景比較

| 場景 | 適合方式 | 原因 |
|------|---------|------|
| 登入功能回歸測試 | 自動化 | 頻繁執行、步驟固定 |
| 新功能探索測試 | 手動 | 需要人類創造力 |
| 資料庫查詢驗證 | 自動化 | 結果可預期、需要大量重複 |
| UI 視覺檢查 | 手動 | 主觀判斷、無法自動化 |
| API 相容性測試 | 自動化 | 可以整合到 CI 管線 |
| 使用性測試 | 手動 | 需要真實使用者回饋 |

## 最佳策略：手動與自動化的平衡

理想的測試策略不是「全部自動化」或「全部手動」，而是在兩者之間取得平衡：

**自動化測試**負責：
- 單元測試（100% 自動化）
- 整合測試（盡可能自動化）
- 回歸測試（自動化）
- API 測試（自動化）

**手動測試**負責：
- 探索性測試
- 使用性測試
- UI 視覺驗證
- 一次性驗證（如上線前 smoke test）

## 小結

手動測試和自動化測試不是非此即彼的選擇，而是互補的策略。手動測試提供人類的洞察力和創造力，自動化測試提供速度、可靠性和可重複性。成功的測試策略是利用兩者的優勢：**讓機器做機器擅長的事（重複、精確、快速），讓人類做人類擅長的事（探索、判斷、創造）**。
