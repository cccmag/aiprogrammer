# 主題六：CAP 定理與一致性

## CAP 定理的背景

CAP 定理由加州大學柏克萊分校的 Eric Brewer 教授於 2000 年提出，是分散式系統設計中最基礎的理論之一。定理指出：分散式系統無法同時滿足以下三個特性：

- **Consistency（一致性）**：所有節點看到相同的資料
- **Availability（可用性）**：每個請求都能獲得回應
- **Partition Tolerance（分割容錯）**：系統在網路分割時仍能運作

## 理解分割容錯

在現實的分散式系統中，網路分割是不可避免的。伺服器可能因為網路問題而無法相互通信，但必須繼續服務。

因此，設計分散式系統時，我們實際上只能在 **一致性** 和 **可用性** 之間做選擇：

- **CP 系統**：放棄可用性，確保一致性（網路分割時拒絕服務）
- **AP 系統**：放棄強一致性，確保可用性（網路分割時仍可服務，但可能返回過時資料）

## NoSQL 與 CAP

不同的 NoSQL 資料庫根據其設計目標，作出了不同的選擇：

### 強一致性系統（CP）

**MongoDB**（單一副本集）：
- 預設寫入需要大多數節點確認
- 網路分割時會停止寫入以保證一致性
- 讀取可以配置為 eventual（最終一致性）

**HBase**：
- 基於 Zookeeper 確保強一致性
- 分割期間會Block寫入

**Cassandra**（可配置）：
- 預設是 AP（最終一致性）
- 但可以配置為 CP（強一致性）

### 可用性優先系統（AP）

**Cassandra**：
- 預設寫入總是成功的
- 使用向量時鐘處理衝突
- 可配置一致性等級（N、W、R 值）

**Redis Cluster**：
- 分割期間從節點可以提供讀取服務
- 但寫入可能丟失

**CouchDB**：
- 離線編輯是設計目標
- 衝突留到之後處理

## 一致性等級

現代分散式資料庫通常提供多種一致性等級，讓應用根據需求選擇：

### 強一致性（Strong Consistency）

所有節點立即看到相同的資料。代價是延遲增加（需要同步確認）和可用性降低（分割時可能不可用）。

```python
# MongoDB 強一致性寫入
result = collection.insert_one(
    {"name": "test"},
    write_concern={"w": "majority"}  # 需要大多數節點確認
)
```

### 最終一致性（Eventual Consistency）

寫入後，資料最終會同步到所有節點，但短期內不同節點可能看到不同資料。延遲低，可用性高。

```python
# Cassandra 最終一致性寫入
session.execute(
    "INSERT INTO users (id, name) VALUES (%s, %s)",
    [user_id, name]
)  # 預設只需要一台節點確認
```

### 因果一致性（Causal Consistency）

保證有因果關係的操作順序一致，沒有因果關係的操作可以並發。

### 讀己之寫一致性（Read Your Own Writes）

確保讀取能看到自己之前的寫入。這是很多應用最基本的需求。

## 實務考量

### 選擇一致性等級

選擇合適的一致性等級需要考慮：

1. **業務需求**：金融交易需要強一致性，社交動態可以容忍最終一致性
2. **網路條件**：網路穩定可以選擇強一致性，不穩定選擇可用性
3. **效能要求**：強一致性通常帶來更高的延遲

### 處理衝突

AP 系統需要面對資料衝突的處理：

```python
# Redis Cluster 的衝突處理
# 最後寫入勝（LWW - Last Write Wins）
# 這是預設策略，可能丟失資料

# 手動衝突解決
def resolve_conflict(local_val, remote_val):
    # 根據業務邏輯合併
    if local_val['timestamp'] > remote_val['timestamp']:
        return local_val
    return remote_val
```

## 超越 CAP

CAP 定理並不是終點。研究者提出了更多細緻的概念：

### PACELC 模型

PACELC（Partition, Availability, Consistency, Else, Latency, Consistency）模型增加了延遲的考量：
- 如果發生分割，選擇可用性還是一致性？
- 如果沒有分割，選擇低延遲還是一致性？

### 柔性事務

一些系統提供「柔性事務」（柔性一致性）：
- **Saga 模式**：將長事務拆分為多個短事務
- **可靠訊息**：確保訊息傳遞最終成功
- **補償機制**：失敗時執行反向操作

## 最佳實踐

1. **理解你的資料**：並非所有資料都需要強一致性
2. **分層設計**：核心資料用強一致性，附屬資料用最終一致性
3. **監控分割事件**：及時發現和處理網路問題
4. **測試失敗場景**：故意製造網路分割來測試系統行為

CAP 定理提醒我們，在分散式系統中，沒有完美的解決方案。理解這些取捨，才能設計出適合特定場景的系統。
