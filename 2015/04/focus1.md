# 主題一：NoSQL 資料庫概述

## NoSQL 的起源

NoSQL（Not Only SQL）這個術語於 2009 年被提出，當時是為了回應傳統關聯式資料庫在處理大規模 Web 應用時遇到的瓶頸。然而，NoSQL 的概念早在 1998 年就由 Carlo Strozzi 提出，他開發了一個不使用 SQL 的輕量級關聯式資料庫。

現代 NoSQL 運動的興起與 Web 2.0 時代的到來密切相關。Facebook、Google、Amazon 等公司面臨處理海量資料的挑戰，傳統的單一伺服器關聯式資料庫已無法滿足需求。

## NoSQL 的分類

NoSQL 資料庫可以分為四大類：

### 鍵值資料庫（Key-Value Stores）

鍵值資料庫是最簡單的 NoSQL 類型，以鍵-值對的形式儲存資料。它們提供極高的讀寫效能，適合用於快取、會話管理和即時計數器等場景。

代表產品：
- **Redis**：支援多種資料結構的記憶體鍵值資料庫
- **Memcached**：純記憶體的分散式快取系統
- **Amazon DynamoDB**：AWS 提供的完全托管鍵值服務
- **Riak**：分散式鍵值資料庫

### 文件資料庫（Document Databases）

文件資料庫以文件為單位儲存資料，通常使用 JSON 或類 JSON 格式。每個文件可以有不同的結構，提供極大的彈性和豐富的查詢能力。

代表產品：
- **MongoDB**：最流行的文件資料庫，提供類 SQL 的查詢語言
- **CouchDB**：Apache 基金會項目，強調離線優先和Replication
- **RethinkDB**：即時推送的文件資料庫

### 列族資料庫（Column-Family Stores）

列族資料庫以列為單位儲存資料，特別適合處理大量寫入和稀疏資料。它們是處理大數據分析工作負載的理想選擇。

代表產品：
- **Apache HBase**：建構在 Hadoop HDFS 之上的分散式資料庫
- **Apache Cassandra**：高度可用的分散式資料庫
- **Google Bigtable**：Google 內部使用的列族儲存

### 圖形資料庫（Graph Databases）

圖形資料庫專門用於處理高度關聯的資料，以節點和邊的形式儲存資料間的關係。它們在社群網路、推薦系統和知識圖譜等場景中表現出色。

代表產品：
- **Neo4j**：最成熟的圖形資料庫
- **OrientDB**：混合文件與圖形的資料庫
- **ArangoDB**：多模型資料庫

## NoSQL 的共同特點

不同類型的 NoSQL 資料庫有一些共同的設計原則：

1. **水平擴展能力**：設計時就考慮了分散式部署，可以通過新增節點來擴展容量
2. **彈性資料模型**：不需要固定的 Schema，可以動態調整資料結構
3. **BASE 特性**：Basically Available（基本可用）、Soft state（軟狀態）、Eventually consistent（最終一致性）
4. **API 導向**：通常使用 RESTful API 或專屬協定進行資料操作

## NoSQL 的適用時機

NoSQL 資料庫並非要完全取代關聯式資料庫，而是在特定場景下提供更好的解決方案。適合使用 NoSQL 的情況包括：

- 需要處理海量資料
- 需要高度彈性的資料模型
- 需要低延遲的即時存取
- 需要橫向擴展能力
- Schema 頻繁變動

## NoSQL 的挑戰

選擇 NoSQL 也意味著面對一些挑戰：

- **查詢能力受限**：不如 SQL 的查詢能力強大
- **最終一致性**：需要接受短期內的資料不一致
- **工具生態**：相比傳統資料庫，工具和成熟度較低
- **學習曲線**：需要學習新的資料建模方式
