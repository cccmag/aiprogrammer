# 主題二：文件資料庫的設計原則

## 文件模型的核心概念

文件資料庫的核心是將資料以文件的形式儲存。文件可以是任何結構化的資料，通常使用 JSON（JavaScript Object Notation）或類 JSON 格式（如 MongoDB 的 BSON）。每個文件都是一個完整的資料單元，包含所有相關的資訊。

與關聯式資料庫的表格（Table）不同，文件資料庫使用集合（Collection）來組織文件。集合的概念類似於表格，但不需要預先定義欄位結構。

## 彈性結構的優勢

文件資料庫最大的特點是結構的彈性。在同一個集合中，不同的文件可以有不同的欄位。這帶來了幾個顯著優勢：

### 快速迭代開發

開發過程中，需求常常會變動。在關聯式資料庫中，改變 Schema 需要執行 ALTER TABLE 語句，可能涉及大量的資料遷移。而在文件資料庫中，只需開始寫入新欄位即可。

### 自然對應物件

文件資料庫的資料模型與物件導向語言中的物件非常接近。開發者可以直接將程式中的物件儲存到資料庫，不需要 ORM（物件關聯對映）的複雜轉換。

### 巢狀資料

文件資料庫允許將相關資料巢狀儲存在同一個文件中。例如，一個部落格文章可以包含作者資訊、標籤和評論，全部儲存在一個文件中，而不是像關聯式資料庫那樣需要多個表格的 JOIN 操作。

## 文件設計模式

### 1. 反正規化設計

在文件資料庫中，反正規化是一種常見的設計策略。為了避免多次查詢，我們可以將經常一起存取的資料放在同一個文件中。

```python
# 反正規化範例：將作者資訊直接嵌入文章文件
article = {
    "_id": "article_001",
    "title": "MongoDB 入門指南",
    "content": "這是一篇關於 MongoDB 的文章...",
    "author": {
        "name": "張小明",
        "email": "zhang@example.com",
        "bio": "資深資料庫工程師"
    },
    "tags": ["MongoDB", "NoSQL", "資料庫"]
}
```

### 2. 引用 vs 嵌入

選擇嵌入還是引用是文件資料庫設計的關鍵決策：

**使用嵌入的時機**：
- 資料是「擁有」關係（如文章和評論，文章擁有評論）
- 資料經常一起存取
- 資料量不大（文件大小建議不超過 16MB）

**使用引用的時機**：
- 資料是「參照」關係（如文章和作者，作者可以被多篇文章引用）
- 資料需要獨立更新
- 資料會被多個文件共享

### 3. 一對多關係

一對多關係在文件資料庫中有多種建模方式：

```python
# 方式一：嵌入（適用於少量子文件）
order = {
    "order_id": "ORD001",
    "customer": "張三",
    "items": [
        {"product": "筆記型電腦", "quantity": 1},
        {"product": "滑鼠", "quantity": 2}
    ]
}

# 方式二：引用（適用於大量或需要獨立存取的子文件）
# 用 ObjectId 引用其他集合的文件
customer = {
    "_id": "ObjectId('...')",
    "name": "張三",
    "email": "zhang@example.com"
}
```

## Schema 設計考量

### 避免過度巢狀

雖然文件資料庫支援巢狀結構，但過度巢狀會導致文件過大，增加讀取複雜度。一般建議巢狀不超過兩到三層。

### 考量未來擴展

設計時應考慮未來可能的擴展需求。預留一些彈性欄位，避免日後需要大規模資料遷移。

### 文件大小控制

大多數文件資料庫對文件大小有限制（MongoDB 限制為 16MB）。設計時應確保文件大小合理。

## 索引設計

文件資料庫通常支援豐富的索引類型：

- **單欄位索引**：對單一欄位建立索引
- **複合索引**：對多個欄位建立索引
- **陣列索引**：對陣列欄位中的元素建立索引
- **文字索引**：對文字內容進行全文搜尋
- **地理位置索引**：支援地理空間查詢

合理設計索引可以大幅提升查詢效能，但過多索引會影響寫入效能。

## 實踐建議

1. **從需求出發**：先了解應用的查詢模式，再設計資料模型
2. **漸進式調整**：資料庫設計不是一成不變的，隨著需求變化適當調整
3. **效能測試**：在設計階段就進行效能測試，確保能滿足效能要求
4. **文件範例化**：建立標準的文件範例，作為團隊設計的參考
