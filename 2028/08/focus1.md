# AI 可觀測性導論

## 從監控到可觀測：2018 到 2028

### 前言

傳統軟體監控（Monitoring）回答「系統出了什麼問題」，而可觀測性（Observability）回答「為什麼系統出了問題」。對於 AI 系統，可觀測性尤為重要——模型的行為無法透過原始碼完全預測，資料分佈隨時可能改變。2020 年後，隨著大規模語言模型進入生產環境，AI 可觀測性成為 MLOps 的核心支柱。

### 可觀測性的三大支柱

可觀測性由三種資料來源構成：

1. **日誌（Logs）**：結構化或非結構化的事件記錄，用於除錯與稽核。
2. **指標（Metrics）**：可聚合的數值（請求量、延遲百分位數、錯誤率），用於趨勢分析與儀表板。
3. **追蹤（Traces）**：端到端的請求路徑，用於分析系統瓶頸。

對 AI 系統，還要加上第四根支柱——**模型行為**（Model Behavior）：輸入分佈、輸出分佈、嵌入空間偏移、預測信心度。

### 傳統監控 vs. AI 可觀測性

```
┌─────────────────┬───────────────────────┐
│   傳統系統       │   AI 系統             │
├─────────────────┼───────────────────────┤
│ 500 錯誤         │ 準確率下降 5%          │
│ CPU 使用率 100%  │ 預測信心度突然降低      │
│ 記憶體用盡       │ 類別分佈開始偏移        │
│ 請求逾時         │ 推理延遲暴增            │
└─────────────────┴───────────────────────┘
```

AI 系統的故障通常不是硬性的「當機」，而是品質的「緩慢退化」，需要統計和機器學習方法來檢測。

### 可觀測性的歷史演進

**2018-2020：基礎設施監控期**。以 Prometheus + Grafana 為代表，監控重點放在 CPU、GPU、記憶體等基礎資源。

**2021-2023：模型監控萌芽期**。AWS SageMaker Model Monitor、WhyLabs、Evidently 等專案出現。開始追蹤特徵分佈和模型效能。

**2024-2026：端到端可觀測性**。OpenTelemetry 成為業界標準，AI 推論流程被納入分散式追蹤。Arize AI、Helicone 等平台崛起。

**2027-2028：AI 驅動的可觀測性**。可觀測性系統本身開始使用 AI 進行根因分析、異常檢測和自動修復。

### 可觀測性成熟度模型

| 等級 | 名稱 | 能力 |
|------|------|------|
| L1 | 無監控 | 系統當了才知道 |
| L2 | 基礎監控 | CPU、記憶體、磁碟 |
| L3 | 模型監控 | 準確率、延遲、漂移 |
| L4 | 端到端追蹤 | 請求層級的可視化 |
| L5 | 自動修復 | AI 驅動的異常檢測與自動回復 |

大部分組織在 2028 年仍處於 L2-L3 階段，而頂尖團隊已邁向 L5。

### 小結

AI 可觀測性不只是技術問題，更是組織能力問題。沒有可觀測性，AI 系統就像一架沒有儀表板的飛機——能飛，但你不知道何時會墜毀。
